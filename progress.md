## 구현한 단계 0 실측 도구

`app/probe/` 아래에 표준 라이브러리 기반 독립 실행 스크립트를 작성했다.

- `run_stage0.py`: 전체, DART 웹 전용, OpenDART 전용 실행 진입점
- `common.py`: 순차 요청, 요청간격 제한, TLS 검증, 응답 저장, API 키 마스킹
- `opendart.py`: 목록 비교, `rm`, 접두어, `013`, D004 실측
- `dart_web.py`: 본문검색 요청, 결과행·0건 마커·구조장애 후보 분류
- `reporting.py`: 통합 결과, 결정사항, 승인 체크리스트 생성
- `README.md`: 실행 방법과 산출물 설명

API 키는 환경변수로만 사용했으며 파일에는 저장하지 않았다. 요청정보는 `tests/fixtures/probe/requests.jsonl`에 `***MASKED***`로 기록한다.

## 실제 실측 결과

### 1. `last_reprt_at=N/Y`

다중 정정이 발생한 실제 사건 3건을 확인했다.

- 한진 `증권신고서(채무증권)`
- 아이엠증권 `주요사항보고서(자본으로인정되는채무증권발행결정)`
- 지에스엔텍 `증권신고서(채무증권)`

세 사건 모두 N에서는 원공시·중간 정정·최종 정정이 반환됐고, Y에서는 해당 사건의 최종 접수만 남았다. 다만 아이엠증권 사례에서 같은 회사·기간·정규화 보고서명에 속하는 별도 사건도 Y 응답에 함께 존재했다. 따라서 회사와 보고서명만으로 정정 사건을 묶으면 안 된다는 결정을 `DECISIONS.md`에 기록했다.

### 2. `rm`의 `정`·`철`

- 서로 다른 사건 30건 검증
- `정` 28건, `철` 2건
- 실제 후속 정정·철회 체인과 30건 모두 일치
- 일치율 100%

### 3. `report_nm` 접두어

다음 공식 접두어 8종을 실제 OpenDART JSON에서 모두 확인했다.

- 기재정정
- 첨부정정
- 첨부추가
- 변경등록
- 연장결정
- 발행조건확정
- 정정명령부과
- 정정제출요구

복수 접두어가 붙는 사례도 확인했다. 예: `[정정제출요구][기재정정]`. `[연장결정]`은 E002가 아니라 B 계열 주요사항보고서에서 관찰되어 탐색 유형을 수정했다.

### 4. OpenDART 상태 `013`

- D004의 `19990101` 하루 기간창에서 실제 `013` 응답 확인
- 메시지: `조회된 데이타가 없습니다.`
- `list`가 없는 정상적인 결과 없음 응답으로 확인
- 장애나 재시도 대상이라는 근거는 발견되지 않음

### 5. D004 공개매수 대상회사

- `corp_name != flr_nm`인 실제 공시 10건 수집
- 원문 ZIP에서 10건 모두 `corp_name`이 대상회사 표기와 일치함을 확인
- 제출인은 `flr_nm`으로 구분됨

### 6. DART 본문검색

- 정상 0건 페이지의 명시적 마커: `조회 결과가 없습니다.`
- 정상 결과 검색어 `주식매수청구권`: 전체 결과 수 8,875건, 표본 결과행 10건 파싱
- 동시성 1로 실행
- 설정 요청간격 1.0초, 실측 최소 요청 시작간격 1.0초
- 마지막 웹 실측 요청 27건 모두 HTTP 200
- 실제 구조장애는 발생하지 않아 `미확인`
- 결과행과 0건 마커가 모두 없는 합성 fixture는 `structure_failure_candidate`로 분류되지만 실제 장애의 증거로 사용하지 않음
- DART `robots.txt`에서 `/dsab007` 명시 금지는 확인되지 않았으나, 자동화 허용 빈도나 명시적 허용 문구도 확인하지 못해 접근정책은 `미확인`

## 계획서와 다른 점 및 결정

`DECISIONS.md`에 다음을 기록했다.

1. 자동화 허용 근거가 추가로 확인될 때까지 DART 본문검색 출시 게이트를 승인하지 않는다.
2. 정정 사건은 회사·정규화 보고서명만으로 묶지 않고 접수번호, 접수일, 정정 순서, 시간 군집을 함께 사용해야 한다.
3. `[연장결정]` 표본 탐색은 E002가 아니라 B 공시유형으로 라우팅한다.
