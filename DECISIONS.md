## 37. 단계 6 S5·S6·S7 온디맨드 관계분석

- 결정: 일반 Fast Path는 계속 접수번호 단위다. 정정·사건 연결은 명시 요청된 S5·S6·S7에서만 실행한다.
- 결정: confirmed 체인은 명시 원접수번호 또는 동일 회사·정규화 보고서명 안에서 유일하게 대응되는 원문 명시 최초제출일을 필요로 한다. 회사·제목·근접일만 같은 항목은 자동 병합하지 않는다.
- 결정: 정정표 행을 구조 diff의 1차 근거로 사용하고 표가 없을 때만 제한된 알려진 필드 정렬을 사용한다. 인식 실패는 `uncertain`/부분비교이며 임의의 변경을 만들지 않는다.
- 결정: `rm=철`은 명시 체인 최종 문서로 연결된 경우에만 철회 confirmed다. `rm=정` 또는 `rm=철`이 남은 최종 후보는 잠정 유효본으로 승격하지 않는다.
- 결정: 공개매수→주식교환 confirmed edge는 날짜순서와 원문 당사자 교집합을 모두 요구한다. 회사명만 같으면 uncertain이다.
- 근거: 단계 0.5 정답 20접수 20/20, 실제 최종 정정본 3유형의 최초제출일·정정표 3/3, 전체 자동시험 154개 통과.

## 36. 단계 5 캐시·OpenDART 동시성 활성화

- 결정: 제한 라이브에서 목록 2가 49.1%, 원문 3이 63.3% 단축되고 오류 0건이어서 v0.2 실행기에 상한 2/3을 연결한다.
- 결정: HTTP 429·일시 timeout·5xx는 원문 동시성을 3→2→1로 낮춘다. OpenDART 020은 요청한도 오류이므로 감속·백오프 대상이 아니며 즉시 중단한다.
- 결정: TTL A~D 비교에서 gzip1이 최초검색 증가 1.703%, 쓰기 p95 6.737ms로 기준을 통과해 `ttl_disk_enabled=true`, 24시간, 500MB, gzip1을 v0.2 기본값으로 한다.
- 결정: `cache_mode=session`은 디스크 계층을 우회한다. 손상 checksum/gzip은 삭제 후 miss이며 원문 ZIP·질의·비밀정보는 캐시하지 않는다.
- 제한: 16요청 활성화 프로브를 장기 p95 또는 모든 검색 90초 보장으로 일반화하지 않는다.

## 35. 단계 3 대화형 MCP 경계

- 결정: 공개 MCP 입력은 `SearchRequest` 단일 계약을 사용하되 단계 3 도구의 `output_mode`는 `interactive`만 노출한다. 공유 모델은 후속 배치 입력을 파싱할 수 있지만 대화형 엔진에서는 반드시 네트워크 전에 `batch_confirmation_required`로 중단한다.
- 결정: `amendment_comparison=true`와 `sequence_required=true`는 현재 Fast Path에서 무시하지 않고 `INTERACTIVE_SCOPE_UNAVAILABLE`로 명시한다. 기능이 구현된 것처럼 보이거나 접수번호 간 추정 연결이 실행되는 것을 막는다.
- 결정: `cache_mode=auto|session`만 허용한다. `temporary_ttl`은 단계 5 성능시험과 활성화 결정 전까지 MCP 계약에서 거절한다.
- 결정: continuation은 opaque token과 프로세스 메모리 상태로 구성하고 기본 TTL 30분·최대 1,000개로 제한한다. 상태에는 lineage, 기간, 목록 창·페이지, 처리 접수번호 해시, 실제 검색어 변형을 저장한다.
- 결정: lineage·기간·검색어 변형이 다른 continuation은 소비하지 않는다. 검증 성공 후에만 토큰을 폐기해 잘못된 후속 요청이 정상 토큰을 파괴하지 못하게 한다.
- 결정: 예산소진과 최신순 편향은 각각 `SEARCH_BUDGET_PARTIAL`, `LATEST_FIRST_BIAS`로 구조화한다. 검색 전 확인·배치승인 상태는 완전한 검색결과가 아니므로 완전성은 `unconfirmed`다.
- 결정: 단계 3의 사건 단위는 접수번호 기반 case 컨테이너다. 서로 다른 접수번호의 사건 병합은 수행하지 않으며 S6·S7의 명시 관계 검증 뒤에만 확장한다.
- 이유: 대화형 MCP는 짧고 예측 가능한 실행, 기계 판독 가능한 부분범위 표시, 사용자 승인 없는 범위확장 방지가 우선이기 때문이다.

## 34. 단계 2 DART 장애 분류·페이지 계약

- 결정: DART HTTP 실패는 공통 HTTP 오류코드를 새 공개 오류코드로 복제하지 않고 어댑터 경계에서 채널 의미로 변환한다. 408·425·429·5xx·연결오류는 `network`, 그 밖의 실제 4xx 거절은 `structure_or_access`다.
- 결정: 단발 네트워크 실패는 회로가 열리지 않았더라도 `OPENDART_TEMPORARY_FAILURE`로 OpenDART 폴백을 허용하고 `blocked_seconds=0`을 반환한다. 두 번째 연속 네트워크 실패는 `DART_FULLTEXT_CIRCUIT_OPEN`으로 승격한다. 명시적 접근거절은 첫 확인에서 바로 15분 회로를 연다.
- 결정: `main.do` 상태진단 성공은 접속 가능성만 확인하므로 이전 `search.ax` 실패횟수를 지우지 않는다. 이후 실제 `search.ax` 성공 또는 차단 만료 후 `PROBING` 성공에서만 회로를 `HEALTHY`로 초기화한다.
- 결정: 회로 이벤트는 절대 차단시각 `blocked_until`과 호환 필드 `blocked_until_epoch`, 실패 분류·횟수·open 횟수를 보존한다. 차단 만료 후 상태진단 결과는 `probe_result`로 별도 기록한다.
- 결정: 페이지 계약은 `ceil(search_count / 10)`, 현재 페이지의 기대 행 수, HTML에 실제 존재하는 `search(n)` 링크의 최댓값을 함께 비교한다. 정상 0건과 범위초과 정상 0건은 변화로 판정하지 않는다.
- 결정: 페이지 계약 변화는 검색을 임의로 계속 완전하다고 주장하지 않고 `PAGINATION_CONTRACT_CHANGED` 경고와 `unconfirmed` 완전성으로 노출한다. 실제 관측 행 수·계산 페이지·링크 마지막 페이지를 진단에 포함한다.
- 이유: 단계 2의 자동 폴백은 장애 원인이 사용자에게 구조화돼야 하고, 단계 0.5에서 실측한 10행 계약이 바뀌면 기존 요청예산·완전성 계산을 그대로 신뢰할 수 없기 때문이다.

실측 결과와 후속 사용자 결정을 기록한다. 미확인 사실과 개인용 사용 결정을 구분한다.

- 결정: 첫 이상 검색응답은 회로 실패가 아니라 후보 상태다. 그 뒤 `main.do` 진단과 동일 `search.ax` 재시도를 순서대로 수행한 뒤 반복될 때만 구조장애를 확정한다.
- 결정: 구조진단 2요청은 안전성 진단용 예약 요청으로 취급하고 일반 결과페이지 예산이 경계에 닿았어도 수행한다. hard deadline은 이 예약 요청에도 그대로 적용한다.
- 결정: 폴백은 결과 건수와 무관하게 최소 `reduced`다. 회로 미개방 폴백의 `blocked_seconds=0`, open 회로는 실제 남은 차단시간의 양의 정수다. hard timeout·continuation이 함께 있으면 `partial`, 필수 검증 실패면 `unconfirmed`가 우선한다.

## 32. 감사 질의 평문 opt-in

- 결정: 기본 감사기록은 원 질의를 저장하지 않는다. 재현용 검색어는 실제 실행한 변형만 기록한다.
- 결정: 평문은 명시적 설정 `audit.audit_query_text=on`에서만 허용하며 기본값은 `off`다. 감사 저장계층은 `query`, `original_query`, `raw_query`, `query_text`, `user_query` 필드를 중첩 위치에서도 제거한다.

## 33. v20에서 발견한 모순과 해석

- DART Fast 예산: §4.3은 Fast DART 요청예산 3회와 모든 상태진단·모드설정·결과·구조재시도를 같은 예산에서 차감하는 공식을 두지만, §4.6·§21.12-6은 최초 상태진단+모드설정+첫 결과 뒤 이상 응답에 `main.do`+`search.ax` 2회를 반드시 요구한다. 최소 5회가 필요해 동시에 만족할 수 없다. 구조 오판 방지가 우선이므로 진단 2회를 예약 요청으로 허용하고 문서에 명시한다.
- Fast Path “다음만 수행”: 단계 1.1 문구를 문자 그대로 전체 검색 동작에 적용하면 v20의 질의변형·후보순위·원문검증 Fast Path와 충돌한다. 이번 구현은 해당 제한을 정정·사건 처리 범위의 allow-list로 해석했고 기존 검색·근거검증은 유지했다.
- `completeness_grade`와 필수채널 실패: §7은 폴백을 최소 `reduced`로 규정하지만 대체 OpenDART 키가 없어 검증 자체가 불가능한 경우는 같은 절의 `unconfirmed` 조건이 더 낮다. 단일 강등 우선순위에 따라 `unconfirmed`를 적용한다.

## 34. 단계 4 승인형 배치 계약

- 결정: 배치 네트워크 실행 권한은 미리보기 자체나 대화형 검색 요청에서 추론하지 않는다. 유효한 프로세스 메모리 `plan_id`, 명시적 `approved=true`, 5/10/15/30분 실행구간이 함께 제시될 때만 실행한다. 거절에는 실행구간 선택을 요구하지 않는다.
- 결정: 같은 질의계보·같은 범위는 기존 plan을 재사용하고 추가 추정요청을 보내지 않는다. 사용자가 거절하면 30분간 같은 계보의 권고를 억제하되 날짜범위·목표량을 합친 작업량이 50% 이상 증가하면 새 미리보기를 허용한다.
- 결정: 체크포인트 단위는 날짜창·공시유형·페이지·페이지 내 행이며 처리 완료 접수번호와 회로상태를 함께 저장한다. 실행구간 soft 경계에서 새 요청을 멈추고, 다음 구간은 별도 승인으로만 재개한다.
- 초기 결정(폐기): 단계 4 승인 배치에 원문 동시성 3을 활성화했다. v21 대조에서 v20 개정요약 6·100과 충돌함을 확인해 결정 35의 순차 1 기준으로 교체했다.
- 결정: 원문 전체와 원 질의 평문은 배치 디스크 자료에 저장하지 않는다. 체크포인트에는 정규화 질의 해시와 실제 검색 변형, 진행상태만 저장하고 결과 레코드는 메타데이터·짧은 근거만 24시간 유지한다. 체크포인트는 재개 가능성을 위해 7일 유지한다.
- 결정: CSV/JSON은 완료 응답만으로 생성하지 않는다. 별도 export 호출에서 사용자가 출력 폴더와 형식을 지정해야 하며, 경로가 없으면 확인요청으로 끝낸다.
- v20 해석: §4.7 예시의 `temporary_ttl_24h`와 단계 4의 “체크포인트 보존기간 표시”는 같은 수명이 아니다. 결과 메타데이터·근거는 24시간, 재개용 체크포인트는 설정 기본 7일로 분리했다. TTL 디스크 원문 캐시는 기본 활성화하지 않는다.
- 초기 해석(폐기): 단계 4의 동시성 3을 승인 배치 상한으로 보았으나, v21에서는 최신 개정요약의 단계 5 이월을 우선한다.

## 35. 단계 2~4 검토의견 판정과 v21 정정

- 반영: CSV 수식 주입 방어, DART 배치 후보발굴, 대화형 배치 권고, `blocked_until·blocked_seconds`, `dart_rate_floor_seconds`, 배치 감사 요약, 실제 export 도구명, 계획 상태, fixture 바이트 고정은 모두 현재 공개 계약 또는 보안 요구와 일치한다.
- 정정: v20 §4.7의 배치 기본 원문 동시성 3 문구와 개정요약 6·100의 “단계 5 전 순차 1”이 충돌한다. 최신 개정요약과 명시적 이월항목을 우선해 운영 동시성은 대화형·배치 모두 1로 유지한다. 기존 결정 34의 배치 동시성 3 활성 해석은 폐기한다.
- 결정: DART 일괄 `search_date_windows`를 그대로 호출하면 구간 hard deadline 중단 시 페이지 위치를 잃는다. 같은 검증된 비중첩 날짜창과 `search_page`를 배치 상태기계에 연결해 날짜창·검색변형·페이지를 원자적으로 체크포인트하는 재개 가능 형태를 사용한다.
- 결정: DART 후보발굴은 정확한 OpenDART 공시유형 필터가 없는 요청에만 우선 적용한다. 공시유형 필터가 있거나 DART 장애가 발생하면 OpenDART 목록 경로를 사용한다. 최종 근거는 어느 경로든 OpenDART 원문으로 검증한다.
- 결정: `export_search_results`를 v21의 정식 명칭으로 채택하고 별도 alias 도구는 추가하지 않는다. 도구 수 증가로 인한 호스트 혼동을 피하고 현재 구현·문서·테스트를 일치시킨다.
- 결정: CSV는 표시값 앞에 작은따옴표를 붙이고 JSON은 원문과 `csv_formula_risk`를 보존한다. hash manifest 대상 파일은 Git의 자동 줄바꿈 변환 대상에서 제외한다.
