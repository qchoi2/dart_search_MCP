# KIND·DART·OpenDART 공시검색 MCP 제작 계획 (v19)

> **v19 개정 요약 — 단계 0.6 실측 반영(검색어 전환·페이지 크기·날짜창·rm 잔여 규칙 확정) (87~92)**
>
> 1. 원문 캐시를 전면 금지하지 않고, **세션 메모리 캐시를 기본 허용**하며 성능시험을 통과한 경우에만 짧은 TTL 디스크 캐시를 활성화한다.
> 2. 표준 검색의 원문 확인 한도를 **최대 40건**, 최종 결과를 **최대 20건**으로 확대한다.
> 3. 대화형 MCP 검색과 별도로 **사용자 승인형 배치 리서치 모드**를 유지한다. 배치 작업은 자동 시작하지 않는다.
> 4. DART 공시통합검색 어댑터가 실패하면 기존 OpenDART 목록·원문검색 방식으로 자동 폴백하고, 구조 변경 또는 접근정책 확인이 필요하다는 경고를 반환한다.
> 5. 원문 전체를 저장하지 않고도 재현성을 확보할 수 있도록 **경량 검색 감사 로그**를 기본 도입한다.
> 6. 외부 요청 병렬성은 목록 2개, 원문 3개를 기본값으로 하고 오류 발생 시 자동 감속한다. 로컬 파싱은 최대 4개까지 허용한다.
> 7. 기존 프로젝트의 기간 분할, 재시도, 인코딩 처리, 원자적 파일 저장, CSV·JSON 출력은 재사용하되, 무기한 원문 캐시와 `get_financials`는 제외한다.
> 8. DART 본문검색 장애 감지, 회로차단기, 중복 사건 묶기, 프롬프트 인젝션 방어, 사용자 승인 토큰 등 운영 안전장치를 추가한다.
> 9. MCP Tasks에 대해서는 특정 미래 사양을 전제하지 않고, 현재 공식 Tasks extension이 실험적이라는 사실만 반영하여 초기 버전에서는 의존하지 않는다.
> 10. DART 본문검색 회로차단기는 모든 장애에 일률적으로 15분을 적용하지 않고, **구조·접근 장애는 15분, 일시적 네트워크 장애는 3분**으로 구분한다. 차단 시 남은 시간과 OpenDART 폴백 사실을 사용자에게 표시한다.
> 11. 배치 리서치 최초 승인 시 사용자에게 **추가 승인을 받을 실행 간격(5·10·15·30분)**을 묻고, 선택값에 따라 구간별 실행·체크포인트·재승인을 수행한다. 기본 권고값은 10분이다.
> 12. “어떠한 내용으로 정정공시된 사례” 검색을 위해 정정 전·후 문서 연결, 정정표 우선 추출 및 구조적 diff 절차를 추가하고, 정정비교 전용 원문 예산을 **최대 80건**으로 확대한다.
> 13. 배치 권고가 과도하게 나타나지 않도록 원시 후보 수가 아니라 **필터 후 예상 원문 수·예상시간·완전성 요구**를 기준으로 단계화하고, 동일 검색계보에서는 권고를 반복하지 않는다.
> 14. 사용자에게 보이는 루트폴더는 **원클릭 설치파일·사용설명서·제거/복구 안내 등 최소 파일만 배치**하고, Python 코드·설정 템플릿·규칙 파일은 `app/` 하위로 이동한다.
> 15. 설치파일 한 번 실행으로 Python 환경 확인, 패키지 설치, OpenDART API 키 입력, `_local_data/.env` 생성, Claude Desktop MCP 설정 병합, 연결검사까지 자동화한다.
> 16. API 키·캐시·로그·배치 결과·체크포인트 등 **배포에 포함하면 안 되는 파일은 루트의 `_local_data/` 폴더에만 저장**하고, 배포 스크립트와 `.gitignore`가 이 폴더를 통째로 제외한다.
> 17. 비전문가용 `사용설명서.html`을 제공하고, OpenDART 가입·API 키 발급·복사 과정까지 쉽게 설명한다.
> 18. 설치파일에는 문서와 돋보기를 든 귀여운 탐정 캐릭터 아이콘을 적용한다.

> 19. 사용자가 검색기간을 명확히 지정하지 않은 경우 임의의 기본기간을 적용하지 않고, 검색 시작 전에 기간을 확인받는다.
> 20. 검색 중 API 키 오류가 의심되면 오류 유형을 구분하여 키 변경·재발급·연결검사 절차를 사용자 언어로 안내한다.
> 21. 검색 완료 후 CSV·JSON·둘 다·파일 불필요 중 원하는 결과파일 형식을 묻고, 사용자가 선택한 경우에만 파일을 생성한다.
> 22. 일반 설치 데이터는 `%LOCALAPPDATA%\공시검색_MCP`에 저장하고, 포터블/개발 배포본에서만 루트 `_local_data/`를 사용한다.
> 23. 전용 런타임, 설치 롤백, 지원환경, DART 본문검색 출시 게이트, 결과 보존, CSV·ZIP·XML 안전장치, 코드서명·업데이트·자체진단 정책을 모두 추가한다.
> 24. DART 본문검색은 **개인용 로컬 사용**을 전제로, 명시적 자동화 허용 근거가 미확인이어도 기술 상태가 정상이고 검색 품질 향상이 확인되면 사용한다. 명시적 금지·접근거부·구조장애가 확인되면 즉시 비활성화하고 OpenDART로 폴백한다.
> 25. `site:kind.krx.co.kr`는 메인 검색기가 아니라 **후보 보강용 보조 검색채널**로 재정의한다. DART 본문검색 결과가 부족하거나, 회사·정확한 공시명을 모르는 사례검색, 거래소 공시 특유 표현 탐색, DART 장애 시 보조 후보 확보에 사용한다.
> 26. 질의 유형·기간·공시유형·예상비용·채널 상태에 따라 DART 본문검색, OpenDART, KIND 검색의 호출순서와 중단조건을 자동 결정하는 **Search Orchestrator**를 도입한다.
> 27. Search Orchestrator를 범용 AI 에이전트가 아니라 **규칙 기반 경량 전략 라우터**로 한정한다. 각 요청에서 10개 전략 중 하나를 우선 선택하고, 결과가 부족할 때만 보조 전략으로 단계적으로 승격한다.
> 28. 오케스트레이터의 품질효과를 검증하기 위해 고정 전략 대비 precision·recall·원문 다운로드 수·첫 결과시간·전체시간을 비교하는 A/B 벤치마크를 도입한다.
> 29. 전략 선택 자체는 로컬 규칙평가로 수행하여 통상 수 밀리초 내에 끝나게 하고, 채널 호출 수를 늘리지 않는 것을 기본 원칙으로 한다.
> 30. 오케스트레이터가 검색시간을 10% 이상 늘리면서 품질 향상이 유의미하지 않으면 해당 전략의 자동승격을 비활성화하거나 규칙을 단순화한다.
> 31. OpenDART 목록의 `rm` 조합문자를 후보 단계에서 해석하되, 후속 정정·철회의 **후보 신호**로만 사용하고 명시적 접수번호·정정사항표·후속 공시로 최종 판정한다.
> 32. 정정 접두어를 공식 `report_nm` 정의에 맞춰 기재정정·첨부정정·첨부추가·변경등록·연장결정·발행조건확정·정정명령부과·정정제출요구로 재분류한다.
> 33. `last_reprt_at` N/Y 실측에서 N은 측정한 3개 사건의 전체 체인, Y는 해당 사건의 최종 접수를 반환했다. 다만 동일 회사·기간·정규화 보고서명에 독립 사건이 함께 존재하므로 Y를 사건 식별키나 유일 최종본의 증명으로 사용하지 않는다.
> 34. OpenDART 상태코드 010·011·012·020·021·800·901을 사용자 조치별로 분리하고, 800은 재시도 없이 서비스 점검 안내로 즉시 중단한다.
> 35. `max_documents` 절대범위 1~120, 정정예산 기본 80·최대 120, `estimated_verified_cases` 모델 필드, 누락 앵커를 명시한다.
> 36. DART 본문검색 초기값은 실측된 **동시성 1·요청 시작간격 최소 1,000ms**로 고정한다. 500ms 또는 동시성 2는 별도 성능·안정성 시험에서 품질 또는 지연 개선이 입증될 때만 개인용 실험값으로 검토한다.
> 37. OpenDART `013`은 오류가 아니라 정상적인 결과 없음으로 처리하고 채널 상태를 `HEALTHY`로 유지한다. `014·100·101·900`도 재시도 여부와 사용자·개발자 조치를 분리한다.
> 38. 목록 요청에는 `sort=date&sort_mth=desc`를 명시하고, 첫 페이지의 `total_count·total_page`를 이용해 호출량·배치 필요성·조기 종료를 실측한다.
> 39. `corp_cls` 사전필터와 S9의 100개 회사 단위 분할을 SearchPlan에 반영한다. 분할 수만큼 예상 호출량을 증가시킨다.
> 40. 공시뷰어 링크는 `https://dart.fss.or.kr/dsaf001/main.do?rcpNo={receipt_no}` 형식으로 고정한다.
> 41. 초기의 빠르고 가벼운 프로그램 원칙을 보존하기 위해 기능을 Fast Path·On-demand·Batch 전용으로 분리하고, 정정 diff·사건연결·KIND·디스크 캐시는 필요한 요청에서만 지연 로딩한다.
> 42. 개정 이력성 보완장은 본문에 계속 누적하지 않고 각 주제 절에 흡수하며, 문서 앞머리에는 최신 변경요약만 유지한다.
> 43. 일반 대화형 검색은 전체 기간창 사전조사를 하지 않되, 이미 조회한 첫 활성 창의 `total_count`를 이용해 남은 범위를 무추가호출로 외삽하고 배치 권고의 1차 근거로 사용한다. 경계값이 불확실할 때만 최대 2개 창을 추가 표본조회한다.
> 44. `SearchPlan`은 실행 전 불변 계약으로 유지하고, `measured_total_count·measured_total_pages·estimation_basis` 등 실행 중 측정값은 `SearchExecutionDiagnostics`로 분리한다.
> 45. 성능목표에서 “첫 후보”와 “전체 검증 완료”를 분리한다. 첫 후보 목표는 목록 기준이며, 25초·60초 예산은 원문 검증을 포함한 전체 검색 예산이다.
> 46. DART 본문검색의 정상 0건 마커 `조회 결과가 없습니다.`를 golden fixture로 고정한다. 결과행도 0건 마커도 없으면 1차로 `structure_failure_candidate`로 분류하고, 상태진단 재시도에서도 반복될 때만 구조장애로 승격한다.
> 47. 세션 메모리 캐시는 40문서 또는 64MB 중 먼저 도달하는 값으로 낮추고, 유휴 메모리와 검색 중 최대 상주메모리 목표를 분리한다.
> 48. 단계 0 실측은 `v0.0-probe`에서 수행했다. OpenDART·정상 0건·동시성 1 기술 게이트는 통과했고, 실제 DART 구조장애와 자동화 허용정책은 미확인으로 남긴다.
> 49. 단계 0 게이트를 OpenDART 목록·정정규칙·D004·DART 파서·DART 정책/구조로 분리하여, DART 정책 미확인이 OpenDART 기반 개발을 막지 않도록 한다.
> 50. 정정 사건은 명시적 원접수번호와 정정사항표를 우선 사용하고, 회사·보고서명·근접기간은 보조수단으로만 사용한다. 불완전한 체인은 `chain_confidence=uncertain`으로 반환한다.
> 51. `rm` 표본 30건은 `정` 28건·`철` 2건이므로 100% 일치율을 전 유형 일반화 근거로 사용하지 않는다. 특히 `철`은 추가 표본과 명시적 연결 근거를 확보한다.
> 52. D004 실측 10건은 5개 `(corp_name, flr_nm)` 조합을 포함하므로 공시 건수와 공개매수 사건 수를 분리하고, 원문 미검증 결과에는 근거와 신뢰도를 표시한다.
> 53. 공식 접두어 8종과 실제 복수 조합 `[정정제출요구][기재정정]`을 규칙파일에 근거 접수번호·표본 수·신뢰도와 함께 저장한다. `[연장결정]`은 B 계열 주요사항보고서로 라우팅한다.
> 54. OpenDART `013` 실측 결론은 목록 API의 정상 무데이터에 한정한다. 다른 엔드포인트와 `014·100·101·900`은 공식 정의·mock·추가 실측의 근거 종류를 구분한다.
> 55. DART Python 3.14 TLS 호환을 위해 필요하면 `VERIFY_X509_STRICT`만 완화하되 호스트명·인증서 체인 검증은 유지하고, `verify=False`는 금지한다.
> 56. 단계 0 raw fixture와 golden fixture를 분리하고, 실행별 `run_id`·요청상한·기한·중단처리·원자적 manifest를 둔다. `__pycache__`와 `*.pyc`는 Git에서 제외한다.
> 57. 단계 0.5에서 플래그십 문구 `상계납입`, `주금납입채무와 상계`, `출자전환` 및 변형어를 실제 DART 본문검색으로 측정하고 형태소·동의어 확장 규칙을 golden fixture로 고정했다.
> 58. DART 본문검색의 `검색건수`, 실효 페이지 크기 10행, `currentPage`, 마지막·빈·범위초과 페이지를 실측하여 충분성·후보예산·배치판정의 1차 신호로 사용한다.
> 59. 유상증자·합병·전환사채의 다중 정정 N/Y 표본을 각각 확보했다. 세 유형 밖으로 일반화할 때는 별도 표본범위와 신뢰도를 표시한다.

# 2. 검색 채널의 역할과 결합 원칙

## 2.1 `site:kind.krx.co.kr` 검색 — 후보 보강용 보조 채널

`site:kind.krx.co.kr` 검색은 메인 검색기가 아니라 다음 상황에서만 선택적으로 사용한다.

* 회사명이나 정확한 공시명을 모르는 상태에서 대표 사례를 빠르게 발굴할 때
* “공개매수 후 포괄적 주식교환”처럼 여러 사건을 연결하는 검색에서 초기 후보 회사를 찾을 때
* DART 본문검색 결과가 부족하여 유사표현·다른 표현을 탐색할 때
* 조회공시, 풍문·보도, 불성실공시, 매매거래정지 등 거래소 공시 특유 표현을 찾을 때
* DART 본문검색 어댑터가 장애 상태일 때 OpenDART 원문검색의 후보를 보강할 때
* 공시 외 기사·보도자료와 함께 거래의 배경을 파악할 때

검색엔진 색인 범위는 완전하지 않으므로 KIND 검색결과만으로 공시의 존재 또는 부존재를 단정하지 않는다. 발견한 모든 후보는 OpenDART 메타데이터 또는 실제 공시 원문으로 검증한다.

KIND 검색은 웹검색 공급자가 사용 가능할 때만 실행한다. MCP 호스트가 웹검색 기능을 제공하면 호스트가 `site:kind.krx.co.kr` 탐색을 담당하고, MCP 서버는 검색문 생성과 OpenDART 검증을 담당할 수 있다. 별도 웹검색 공급자가 없으면 KIND 채널이 없어도 핵심 검색은 DART 본문검색과 OpenDART 폴백만으로 동작해야 한다.

## 2.2 DART 공시통합검색(본문검색)

희귀 문구 및 거래구조 탐색의 우선 채널로 사용한다.

주요 용도:

* `상계납입`, `주금납입채무와 상계` 등 본문 문구 검색
* 동일 개념의 표현 변형별 후보 발굴
* 접수번호·회사명·공시명을 확보한 뒤 OpenDART 검증으로 연결

주의사항:

* 공식 OpenAPI가 아닌 웹 화면용 검색 기능이므로 HTML, 요청 파라미터 또는 세션 구조 변경에 취약할 수 있다.
* DART의 이용약관, 정보이용 유의사항 및 자동화 접근 허용 범위를 단계 0에서 확인한다. 명시적 허용 빈도는 확인하지 못했으나, 본 프로젝트는 개인용 로컬 사용이라는 사용자 결정을 적용한다.
* 초기 호출은 동시성 1·요청 시작간격 최소 1,000ms로 순차 실행하고 회로차단기를 적용한다.
* 형태소 분석과 동의어 처리 특성은 문서화된 보장사항으로 간주하지 않고 기지 정답 세트와 질의별 검색건수·신규 접수번호 기여도로 계속 회귀 측정한다.

실측된 본문검색 요청 계약은 쿠키를 유지한 상태에서 `/dsab007/detailSearchMain2.do`로 검색모드를 설정한 뒤 `/dsab007/search.ax`로 결과를 요청하는 흐름이다. 단계 0.6에서 동일 쿠키 세션·동일 검색 모드의 검색어 전환 3회를 시험한 결과, 검색어가 바뀌어도 모드설정 POST 없이 `/search.ax`만 호출한 결과가 재설정 대조군과 정확히 일치했다. 따라서 세션 또는 검색 모드가 바뀔 때만 모드설정을 수행하고 같은 모드의 검색어 변형은 직접 조회한다. `Referer`와 `X-Requested-With`를 포함하고, `검색건수` 전체값과 현재 페이지의 접수번호 결과행 수를 구분한다. DART `robots.txt`에서 자동화 대상 `/dsab007`의 명시적 금지는 관찰되지 않았지만, 뷰어 `/dsaf001/main.do`는 금지 경로이므로 프로그램이 자동 수집하지 않고 사용자 링크 제공에만 사용한다.

단계 0.5의 2025-07-16~2026-07-16 검색창에서 플래그십 문구를 실측했다. 붙여쓴 `상계납입`은 42건, `주금납입채무와 상계`는 54건으로 정밀 기본질의에 사용한다. 띄어쓴 `상계 납입`은 50,864건, `주금 납입 채무와 상계`는 531건이었고 전자는 단순 회계상 상계와 서로 떨어진 토큰 매칭이 대량 포함되므로 broad 확장에서만 사용한다. `출자전환`은 13,084건이며 `채권의 출자전환`은 7,228건이었지만 첫 페이지 신규 접수번호 기여가 0이므로 후자를 기본 변형 묶음에서 제외하고 사용자가 채권 표현을 명시하거나 broad 확장이 필요할 때만 사용한다.

첫 페이지 기준 원문 6건에서 직접 상계납입 4건과 출자전환 2건을 확인했다. 이 표본은 검색 가능성과 기본 질의 우선순위의 근거이지 시장 전체 precision·recall이나 영구적인 검색건수의 보장은 아니다. `measured_window`, `measured_at`, `search_count`, 첫 페이지 중복률과 신규 후보 기여도를 golden fixture와 진단정보에 함께 저장하며 DART 색인 변화로 수치가 달라져도 파서 실패로 오판하지 않는다.

실측된 결과행에는 시장문자(유·코·기 등), 공시유형 그룹 태그(`[발행공시]`, `[주요사항보고]` 등), `[본문]`/`[첨부문서]` 구분, 제출인, 접수일이 포함된다. 동일 접수번호가 본문 행과 첨부문서 행으로 반복되는 것이 페이지 내 중복의 구조적 원인이므로, 이 태그들을 구조 필드로 파싱해 중복제거를 표본 추정이 아닌 결정적으로 처리하고 본문 매치를 첨부 매치보다 우선 검증한다. 행의 유형 태그·접수일은 OpenDART 조회 전 사전필터로 사용할 수 있다. 또한 실측 form의 정렬은 `sort=DATE`(내림차순)뿐이며 관련도 정렬 옵션은 관찰되지 않았다. 따라서 조기 종료는 최신순 편향을 가지며 이를 진단정보에 표시하고, 이 문서에서 “관련도”는 서버 정렬이 아니라 로컬 mechanical_score를 의미한다. DART 검색화면의 정보이용 유의사항은 검색 정확도를 보장하지 않는다고 명시하므로, 본문검색 단독 근거 결과에는 이 한계를 완전성 등급과 안내문에 반영한다.

## 2.3 OpenDART API

OpenDART는 다음 데이터의 기준 소스로 사용한다.

* 회사 고유번호 및 회사명
* 공시 제출회사와 제출인
* 보고서명, 접수일, 접수번호
* 공시유형 및 상세유형
* `rm` 비고 플래그에 따른 후속 정정·철회·시장소관·연결보고 여부
* `report_nm` 공식 접두어에 따른 정정·첨부변경·확정·감독당국 조치 구분
* 정정·철회·최종보고서 여부
* 공시 원문 ZIP/XML

회사 고유번호가 없는 전체시장 검색은 OpenDART 제한에 맞춰 최대 3개월 구간으로 분할하고, 페이지당 결과 수는 100건으로 설정한다. 특정 회사의 `corp_code`가 있으면 회사 기준 검색을 우선한다.

### OpenDART 목록 필드의 선행 활용

원문을 다운로드하기 전에 목록 응답의 다음 필드를 최대한 활용한다.

* `rm`: 조합문자를 `rm_flags[]`로 정규화한다. 단계 0에서 `정` 28건과 `철` 2건은 후속 정정·철회 문서와 일치했다. 단계 0.5에서 `유=유가증권시장`, `코=코스닥시장`, `넥=코넥스시장`, `공=공정거래위원회`, `연=연결`은 공식 의미와 각 5개 실표본을 확보해 `confirmed`로 승격했다. 단계 0.6에서는 실제 `rm=채` 16건을 확보해 `채=채권상장법인` 의미를 `confirmed`로 승격했다. 다만 모두 단일문자여서 `채` 포함 조합의 실제 순서·다른 플래그 보존은 `unconfirmed`다.
* `report_nm`: 공식 접두어를 분리하여 본문 기재변경, 첨부변경, 첨부추가, 조건확정, 감독당국 조치 등을 구분한다.
* `corp_name`과 `flr_nm`: 공개매수(D004)에서 두 값이 다르면 `corp_name`을 대상회사 1차 식별값으로 사용한다.

`rm`은 후속 문서의 존재를 알려주는 선행 신호이지 정정 체인 전체를 직접 제공하는 필드는 아니다. 단계 0.6의 `철` 재감사에서는 후속 문서가 원 제출일 또는 유일한 상품·주제를 명시한 독립 표본 9건을 확보했지만, 같은 회사·제출인·근접일의 철회보고서가 실제로는 다른 원계획을 가리키는 반례도 확인했다. 따라서 회사·정규화 보고서명·근접기간만으로 사건을 합치거나 제외하지 않는다. S6·S7에서는 후보 우선순위와 불필요한 다운로드 축소에 사용하되, 최종 유효본·철회여부는 명시적 원접수번호, 원 제출일, 정정사항표 및 관련 접수문서로 검증한다.

`rm_flags[]` 파서는 알려진 단일문자 토큰을 원문 순서대로 모두 분리하고 원시 `rm`도 보존한다. 단계 0.5에서 관찰한 `유정`, `코정`, `넥정`, `정연`, `공정`을 회귀 fixture로 고정한다. 예를 들어 `공정`은 하나의 미정의 단어가 아니라 `[공, 정]`이며, 알 수 없는 문자가 섞이면 버리지 않고 `unknown_rm_flags[]`에 남긴다. 조합에 `정`이 포함돼도 시장 플래그를 잃지 않으며, 반대로 시장 플래그만으로 후보를 하드 제외하지 않는다.

### 대화형 검색에서 검색기간 확인 및 3개월 제한을 처리하는 방식

사용자가 검색기간을 명확히 지정하지 않은 경우에는 임의로 최근 3년·5년·10년 등을 적용하지 않는다. 검색 시작 전에 다음처럼 기간을 확인받는다.

```text
검색기간이 지정되지 않았습니다. 어느 기간의 공시를 검색할까요?
예: 2020-01-01 이후 / 최근 5년 / 2015-01-01~2026-07-16
```

다만 사용자의 표현에서 기간이 객관적으로 특정되는 경우(예: “2025년”, “최근 1년”, “공개매수 이후”)에는 그 범위를 해석해 표시하고, 모호성이 있으면 확인받는다. 기간 확인 전에는 대량 API 호출을 시작하지 않는다.

OpenDART의 3개월 제한은 우회하는 것이 아니라 MCP가 사용자에게 보이지 않게 **연속된 달력 기준 3개월 창으로 분할·병합**한다.

1. 특정 회사의 `corp_code`가 있으면 회사 기준 검색을 사용하여 불필요한 기간분할을 피한다.
2. `corp_code`가 없고 검색기간이 3개월을 초과하면 `[시작일, 시작일+3개월-1일]` 형태의 겹치지 않는 기간창을 만든다.
3. 대화형 검색은 최신 사례를 우선 찾기 위해 기간창을 최신순으로 처리한다. 목록 기간창은 최대 2개까지만 병렬 실행한다.
4. 각 기간창은 `page_count=100`으로 조회하고, 결과가 여러 페이지이면 해당 창 안에서 페이지네이션한다.
5. 모든 창의 결과를 접수번호로 중복제거한 뒤 공시일·접수번호 기준으로 정렬한다.
6. DART 본문검색에서 접수번호 후보를 직접 확보한 경우에는 전체 기간창을 모두 조회하지 않고 해당 후보만 OpenDART로 검증한다.
7. 대화형 호출예산 또는 90초 하드 시간예산에 도달하면 처리한 기간범위를 명시하고 continuation token을 반환한다.

예를 들어 2024-01-01부터 2026-07-16까지 전체시장을 검색하면 약 11개의 기간창으로 나뉜다. 대화형 검색은 최신 창부터 필요한 만큼만 처리하고 목표 결과를 확보하면 조기 종료한다. 전체 11개 창을 빠짐없이 처리해야 하는 요청은 예상시간을 계산한 뒤 필요할 때만 승인형 배치로 전환한다.

### 목록 요청 정렬·사전필터·실측 예산

* 모든 목록 요청은 `sort=date&sort_mth=desc`를 명시한다. API 기본값에 의존하지 않는다.
* 첫 페이지 응답의 `total_count`와 `total_page`를 기록한다. 이후 호출량은 추정치보다 이 실측값을 우선 사용한다.
* 목표 결과 수와 현재 검증수, 페이지 정렬이 조기종료를 허용하면 남은 페이지를 조회하지 않는다.
* 사용자가 단일 시장을 지정하면 `corp_cls`를 사전필터로 사용한다: `Y` 유가증권, `K` 코스닥, `N` 코넥스, `E` 기타. 복수값을 한 요청에 넣을 수 없으므로 복수시장 요청은 별도 호출로 분할한다.
* 창별 첫 페이지 사전조사는 전수성·배치 요청에서만 수행한다. 일반 대화형 검색에서는 첫 활성 창부터 처리하여 불필요한 선행 호출을 만들지 않는다.
* 일반 대화형 검색은 이미 조회한 첫 활성 창의 `total_count`와 현재 필터 통과율·원문 검증률을 이용해 남은 기간의 문서수를 외삽한다. 이 계산은 추가 API 호출을 만들지 않으며 `diagnostics.estimation_basis=extrapolated_first_window`로 표시한다.
* 외삽값이 40·80건 또는 5분 배치 경계의 ±20% 범위에 있어 판정이 민감한 경우에만 최대 2개의 인접 기간창 첫 페이지를 추가 표본조회한다. 이때 `estimation_basis=sampled_windows`와 표본 창 수를 기록한다.
* 정기공시 집중시기 등 계절성 편차를 고려하여 외삽값은 확정치가 아니라 배치 권고용 추정치로만 사용한다.
* 배치 미리보기는 각 기간창 첫 페이지의 `total_count·total_page`, 회사분할 수, 예상 원문검증률을 조합해 `estimated_calls`와 `estimated_verified_cases`를 계산한다.
* 단계 0 discovery 실측(2026-07-01~07-16 및 2026-04-01~06-30 창의 공시유형별 페이지 수)을 `volume_priors.yaml`에 유형·기간 밀도 prior로 저장한다. 첫 활성 창을 조회하기 전의 초기 추정에는 이 정적 prior를 사용하고, 첫 페이지 실측값이 확보되면 즉시 대체한다. prior에는 `measured_window`와 낮은 신뢰도를 명시하며 추가 API 호출을 만들지 않는다.

## 2.4 Search Orchestrator — 경량 전략 라우터

### 2.4.1 도입 필요성에 대한 종합 판단

오케스트레이터는 도입할 필요성이 인정된다. 이유는 검색 채널마다 강점과 비용구조가 달라 단일 고정 순서로 처리하면 다음 문제가 반복되기 때문이다.

* 특정 회사의 단순 목록검색에도 DART 본문검색과 KIND를 호출해 불필요한 지연이 생길 수 있음
* 희귀 문구 검색을 OpenDART 목록부터 시작하면 원문 다운로드가 폭증할 수 있음
* 정정공시 비교를 일반 검색 한도로 처리하면 사건 수가 급감함
* 공개매수 후 주식교환처럼 사건 연결형 질문을 단순 키워드검색으로 처리하면 오탐이 증가함
* DART 본문검색 장애 중에도 같은 채널을 반복 호출하면 타임아웃이 누적됨

반대로 오케스트레이터가 각 검색마다 모든 채널을 순차 또는 병렬 호출하면 품질보다 비용이 더 커질 수 있다. 따라서 본 프로젝트에서는 오케스트레이터를 **별도 LLM을 호출하는 범용 에이전트가 아니라, 입력 스키마와 명시적 규칙으로 동작하는 경량 상태기계**로 구현한다.

예상 효과는 다음과 같다.

* 적절한 1차 채널 선택으로 불필요한 원문 다운로드 감소
* 질의 유형별 검증절차 적용으로 precision 향상
* 결과 부족 시에만 보조채널을 사용하여 recall 보완
* 채널 장애 시 즉시 폴백하여 반복 타임아웃 감소
* 검색결정 로그를 남겨 재현성과 디버깅 가능성 향상

전략 선택은 문자열·회사·기간·공시유형·완전성 요구·채널 상태를 평가하는 로컬 규칙만 사용하므로, 오케스트레이터 자체의 CPU·메모리 오버헤드는 미미해야 한다. 실제 속도를 좌우하는 것은 외부 검색 및 원문 다운로드이며, 오케스트레이터는 이를 늘리는 것이 아니라 줄이는 방향으로 설계한다.

### 2.4.2 처리 단계

```text
SearchRequest 정규화
→ 필수조건 확인(특히 검색기간)
→ 10개 전략 중 1차 전략 선택
→ SearchPlan 생성
→ 1차 채널 실행
→ 후보 품질·수량·커버리지 평가
→ 충분하면 종료
→ 부족하면 사전에 허용된 보조전략 1개만 승격
→ 예산 초과 예상 시 부분결과 또는 배치 미리보기
```

한 요청에서 활성 전략은 원칙적으로 하나이며, 보조전략 승격은 최대 2단계로 제한한다. 모든 전략을 동시에 실행하지 않는다.

### 2.4.3 공통 입력 스키마

```json
{
  "query": "유상증자 공시 중 상계납입 사례",
  "company": null,
  "date_from": "2020-01-01",
  "date_to": "2026-07-16",
  "target_count": 10,
  "mode": "standard",
  "max_documents": null,
  "cache_mode": "auto",
  "exhaustive": null,
  "amendment_comparison": false,
  "sequence_required": false,
  "requested_output": "interactive"
}
```

검색기간이 불명확하면 전략 선택 전에 사용자에게 확인한다.

### 2.4.4 10개 검색전략 템플릿

#### S1. `company_filing_first` — 특정 회사·공시유형 검색

적용 예: “삼성전자의 최근 유상증자 관련 공시를 찾아줘.”

```text
회사 식별 → corp_code 기준 OpenDART 목록 → 보고서명/유형 필터 → 필요한 원문만 검증
```

KIND와 DART 본문검색은 기본 생략한다.

#### S2. `company_fulltext_first` — 특정 회사·본문주제 검색

적용 예: “A사 공시에서 원주 전환 관련 내용을 찾아줘.”

```text
회사명+본문문구 DART 검색 → 접수번호 후보 → OpenDART 검증 → 부족 시 회사별 목록 원문검색
```

#### S3. `market_fulltext_first` — 시장 전체 희귀 문구 검색

적용 예: “상계납입 사례를 찾아줘.”

```text
DART 본문검색 변형어 → 후보 정규화 → 상위 후보 원문 검증 → 부족 시 KIND 보강 → 마지막으로 OpenDART 기간분할 폴백
```

단계 0.5에서 S3의 플래그십 문구 6종과 원문 6건을 실측해 이 전략을 활성화했다. 정밀 기본질의는 `상계납입`과 `주금납입채무와 상계`이며, `출자전환`은 상계납입 질의의 보조 개념질의로 사용한다. `상계 납입`과 `주금 납입 채무와 상계`는 broad 확장에만 사용한다. 특히 `상계 납입`은 50,864건과 높은 접수번호 중복을 보여 기본 자동확장에 넣지 않는다. `채권의 출자전환`은 `출자전환` 대비 첫 페이지 신규 기여가 0이므로 기본 묶음에서 제외하고 사용자 문구 또는 broad 단계에서만 호출한다.

각 질의는 `검색건수`, 측정기간, 첫 페이지 접수번호, 페이지 내·페이지 간 중복률, 이전 변형까지의 누적집합 대비 신규 접수번호 기여도를 저장한다. 검색어 정밀화를 페이지 깊이 확대보다 먼저 수행하고, 변형어가 신규 검증사례를 제공하지 않거나 거짓양성만 크게 늘리면 자동확장 목록에서 제거한다. 모든 변형의 결과는 전역 `rcept_no` 집합으로 중복제거한 뒤 원문 예산을 사용한다.

#### S4. `filing_type_first` — 구조화된 공시유형 중심 검색

적용 예: “공개매수 공시 사례 20건.”

```text
OpenDART 상세유형/공시유형 → 회사·기간 필터 → 정정/철회 정리 → 원문 검증
```

#### S5. `event_sequence` — 복수 사건 선후관계 검색

적용 예: “공개매수 후 포괄적 주식교환 사례.”

```text
선행사건 후보 → 대상회사 식별 → 후행사건 검색 → 날짜·당사자 관계 검증 → 사건 단위 결과
```

#### S6. `amendment_compare` — 정정 전후 내용 비교

적용 예: “납입일이 연기되도록 정정된 사례.”

```text
정정공시 후보 → 정정 체인 연결 → 정정표 우선 추출 → 구조적 diff → 변경유형 판정
```

정정비교 전용 원문 예산 최대 80건을 적용한다.

#### S7. `latest_effective_filing` — 최신 유효본 확인

적용 예: “이 거래의 최종 정정공시가 무엇인지 확인해줘.”

```text
사건 후보 → 원공시/정정/철회 체인 → 최종 유효본 판정 → 변경이력 요약
```

#### S8. `exchange_disclosure_first` — KIND·거래소 특유 공시 검색

적용 예: “조회공시 요구와 답변 사례”, “매매거래정지 공시.”

```text
OpenDART 거래소공시 유형 및 KIND 후보검색 → 메타데이터 검증 → 관련 원문 확인
```

KIND는 공급자 또는 호스트 웹검색이 있을 때만 사용한다.

#### S9. `cross_company_precedent` — 유사 거래 선례 탐색

적용 예: “A사 거래와 유사한 공시 사례 10건.”

```text
기준 사건 특징 추출 → 공시유형·본문표현 조합검색 → 대상회사 집합 계산
→ 회사가 100개를 넘으면 100개 이하 묶음으로 분할
→ 회사 다양성 필터 → 원문 근거 검증
```

한 회사의 반복 정정본이 결과를 잠식하지 않도록 사건 단위로 묶는다. OpenDART의 회사수 제한을 넘는 경우 `company_batch_count = ceil(company_count / 100)`으로 계산하고, `estimated_calls`와 시간예산에 해당 배수를 반영한다. 대화형 예산을 넘으면 일부 묶음만 실행하고 continuation 또는 배치 미리보기를 반환한다.

#### S10. `exhaustive_batch_candidate` — 전수성·대량조사 판정

적용 예: “2015년 이후 가능한 모든 상계납입 사례를 빠짐없이 찾아줘.”

```text
예상 기간창·후보·원문 수 산정 → 대화형으로 가능한 부분 먼저 판단 → 필요 시 배치 미리보기 → 사용자 승인 후 구간 실행
```

이 전략은 검색을 자동 시작하지 않고 승인계획만 생성한다.

### 2.4.5 전략 선택 우선순위

```text
전수·빠짐없이 명시                     → S10
정정 전후 변경내용 요청                → S6
최종 유효본·정정이력 요청              → S7
복수 사건의 선후관계 요청              → S5
거래소 특유 공시                       → S8
특정 회사 + 명확한 공시유형            → S1
특정 회사 + 본문 주제                  → S2
명확한 공시유형의 시장 사례            → S4
그 밖의 희귀 본문 문구                  → S3
유사 선례·비교사례                      → S9
```

이 순서는 17.4절의 최초 일치 규칙과 동일하며, 두 절이 어긋나면 17.4절을 기준으로 정정한다. 둘 이상의 조건이 충돌하면 더 구체적인 전략을 우선한다. 예를 들어 특정 회사의 정정 전후 비교는 S1이 아니라 S6을 선택하되 회사 필터를 추가한다.

### 2.4.6 충분성 판정과 단계적 승격

다음 조건을 모두 만족하면 검색을 종료한다.

* 확정 결과 수가 사용자의 목표 수에 도달
* 각 결과에 OpenDART 메타데이터 또는 원문 근거가 존재
* 정정·철회·중복 사건 처리가 완료
* 실제 검색기간과 채널 커버리지를 설명할 수 있음

부족한 경우에만 전략에 지정된 보조경로를 실행한다.

DART 본문검색을 사용한 경우 `검색건수`와 현재 페이지 결과행 수를 충분성 판정에 함께 사용한다. `검색건수`가 많다는 이유만으로 자동 배치 전환하지는 않지만, 다음을 계산하는 1차 신호로 사용한다.

* `effective_page_size`: 실제 접수번호 결과행 수. 단계 0.6에서 form의 허용값 15·30·50·100과 `maxResultsCb=100`까지 시험했지만 모두 10행이므로 운영 기본값은 10
* `estimated_result_pages = ceil(search_count / effective_page_size)`
* `dedupe_yield_rate`: 조회한 페이지의 전체 행 중 전역 중복제거 후 새 접수번호 비율
* `estimated_unique_receipts_per_page = max(1, effective_page_size * dedupe_yield_rate)`
* `estimated_unique_result_count = ceil(search_count * dedupe_yield_rate)`
* `candidate_pages_needed = ceil(min(estimated_unique_result_count, candidate_cap) / estimated_unique_receipts_per_page)`
* `estimated_dart_requests`: 검색모드 설정·첫 페이지·추가 페이지를 합산한 요청 수

`search_count`와 `effective_page_size`는 결과행 기준이며 고유 공시 수가 아니다. 단계 0.5의 넓은 질의에서는 같은 접수번호가 한 페이지 안과 1·2페이지 사이에서 반복됐다. 따라서 페이지와 변형어 전체에 걸쳐 `rcept_no`를 중복제거하고, 최초 1~2페이지의 `dedupe_yield_rate`로 고유 후보 수를 보수적으로 추정한다. 초기 페이지 표본이 전체 깊이를 대표한다는 보장은 없으므로 `dedupe_estimation_basis=sampled_first_pages`와 낮은 신뢰도를 함께 기록하고 실제 페이지를 읽을 때마다 갱신한다.

남은 DART 결과페이지 호출예산을 `available_result_page_calls`로 계산한다. `estimated_result_pages <= available_result_page_calls`이면 사용자가 요구한 범위의 페이지 순회를 허용한다. 초과하면 먼저 붙여쓴 표현·긴 구문·회사·공시유형으로 질의를 정밀화하고, 그래도 전수 완주가 필요하면 단계 0.6에서 검증한 포함 경계의 비중첩 연속 날짜창으로 검색기간을 나눈다. 창별 완료와 기간 연속성을 확인하고 전역 `rcept_no` 합집합으로 병합한다. 대표 사례 검색은 정밀 질의의 상위 페이지에서 목표 건수를 확보하면 조기 종료하되, 실측 정렬이 날짜 내림차순이므로 조기 종료 결과의 최신순 편향을 coverage에 표시한다. 전수성 요구이거나 정밀화·날짜창 분할 후에도 완주할 수 없으면 S10·배치 미리보기로 전환한다. 페이지를 깊게 읽는 것보다 질의 정밀도를 먼저 높인다.

```text
1차 결과 충분              → 즉시 종료
수량 부족                  → 정밀 변형 또는 보조채널 1회
근거 부족                  → 상위 후보 원문 추가 검증
페이지 완주 불가           → 질의 정밀화, 대표검색 조기종료 또는 배치
시간/호출예산 근접         → 부분결과 + continuation
전수성 요구 또는 5분 초과  → 배치 미리보기
```

### 2.4.7 성능 보호장치

* 전략 선택 로직에는 네트워크 호출을 넣지 않는다.
* 한 요청의 자동 전략 승격은 최대 2회로 제한한다.
* KIND를 포함한 보조채널은 결과 부족 시에만 호출한다.
* 동일 접수번호는 어느 전략에서 발견되더라도 한 번만 원문 다운로드한다.
* 후보 수집과 원문 검증을 분리하고, 원문은 로컬 기계적 점수(mechanical_score) 상위 순으로 조기 종료한다.
* 전략별 요청·문서 예산을 SearchPlan에 명시한다.
* 예상 원문 수가 대화형 한도를 크게 초과하면 실행 전부터 S10으로 전환한다.
* 오케스트레이터 결정로그는 검색 종료 시 한 번만 JSONL에 기록한다.

### 2.4.8 품질·성능 검증 게이트

오케스트레이터 도입효과는 추정만으로 확정하지 않고 프로토타입 벤치마크에서 검증한다.

비교 대상:

1. 고정 순서 방식: DART 본문검색 → OpenDART → KIND
2. 10개 전략 오케스트레이터 방식

측정 항목:

* 기지 정답 세트 대비 recall
* 최종 20건 중 실제 관련 사례 비율(precision)
* 첫 검증 결과 반환시간
* 전체 검색시간
* OpenDART 목록 호출 수
* DART 본문검색 호출 수
* 원문 다운로드 수
* KIND 보조채널 호출률
* 배치 권고율
* 잘못된 전략 선택률

출시 기준 권고값:

```text
recall 또는 precision이 고정 방식보다 5%p 이상 개선
또는 동일 품질에서 원문 다운로드/외부호출이 20% 이상 감소
오케스트레이터 적용 후 일반 검색 p95 시간 증가 10% 이하
전략 선택 자체 p95 50ms 이하
잘못된 전략 선택률 5% 이하
```

위 기준을 만족하지 못하는 전략은 자동 라우팅에서 제외하거나 규칙을 단순화한다. 전체 오케스트레이터가 품질·비용 어느 쪽에서도 개선을 보이지 않으면 고정 전략으로 폴백할 수 있도록 feature flag를 둔다.

## 2.5 기본 검색 흐름

```text
사용자 질의 분석
→ 검색 범위·공시유형·표현 변형 계획 수립
→ DART 본문검색 또는 OpenDART 유형검색으로 후보 수집
→ OpenDART 메타데이터 확인
→ 후보 중복·정정본·동일 사건 묶기
→ 필요한 원문만 다운로드·검증
→ 기계적 매칭 사실과 법률적 판단을 분리
→ 최대 20건 결과와 검색범위·진단정보 반환
```

질의별 우선순위:

* 특정 회사의 공시 목록: OpenDART 우선
* 희귀 문구 및 비정형 거래구조: DART 본문검색 우선
* 공개매수 등 상세유형이 명확한 공시: OpenDART 유형검색 우선
* DART 본문검색 장애: OpenDART 목록·원문검색으로 자동 폴백

## 2.6 DART 본문검색 실패 시 자동 폴백

다음 상황을 **어댑터 실패**로 판단한다.

* 반복적인 HTTP 오류 또는 타임아웃
* 예상 HTML 요소나 필수 응답 필드가 사라짐
* 로그인·세션·보안페이지 등 검색결과가 아닌 화면이 반환됨
* 파서가 검색결과 페이지의 구조를 인식하지 못함
* 사전에 정의한 저비용 상태확인 요청이 반복 실패함

정상적인 검색 결과가 0건이라는 사정만으로 장애로 보지는 않는다. 다만 검색 어댑터 상태확인도 실패하거나 응답 형식이 비정상인 경우에는 폴백한다.

폴백 절차:

```text
DART 본문검색 실패
→ 해당 검색 채널 회로차단기 활성화(장애유형별 3분 또는 15분, 4.6절)
→ OpenDART 목록 API로 공시유형·기간별 후보 수집
→ 보고서명 및 검색어 규칙으로 후보 축소
→ 예산 범위 내 원문 다운로드·문구 검색
→ 결과에 폴백 경고 포함
```

반환 경고 예시:

```json
{
  "code": "DART_FULLTEXT_FALLBACK",
  "message": "DART 본문검색 어댑터가 정상 작동하지 않아 OpenDART 원문검색으로 전환했습니다. DART 검색 화면의 구조 변경 여부, 접근정책 및 이용약관을 확인하세요.",
  "fallback_source": "opendart_document_search"
}
```

폴백이 실행되면 결과의 완전성 등급을 한 단계 낮추고, 실제로 원문을 확인한 문서 수와 검색하지 못한 잔여 후보 수를 표시한다.

# 3. 저장·캐시·검색 기록 정책

## 3.1 기본 저장 원칙

기본적으로 전체 공시 메타데이터, 원문 저장소, 전문검색 인덱스 또는 정기 증분수집을 구축하지 않는다.

```text
storage_mode = ephemeral
```

기본 모드에서는:

* 사용자가 요청하지 않은 회사나 기간을 미리 수집하지 않는다.
* 검색 결과를 공시 코퍼스로 누적하지 않는다.
* 전문검색용 DB나 벡터 DB를 자동 구축하지 않는다.
* 배치 리서치 또는 영구 인덱스는 사용자 승인 없이 시작하지 않는다.

## 3.2 원문 세션 캐시와 짧은 TTL 캐시

검색속도 개선을 위해 다음 캐시 계층을 허용한다.

### A. 세션 메모리 캐시 — 기본 활성화

```text
cache_mode = session
```

* 같은 MCP 프로세스가 살아 있는 동안 접수번호별 파싱 텍스트를 메모리에 재사용한다.
* 디스크 쓰기가 없으므로 최초 검색속도를 거의 저하시키지 않는다.
* LRU 방식으로 제한하고, 기본 상한은 40개 문서 또는 64MB 중 먼저 도달하는 값으로 한다. 파싱 텍스트의 실제 크기를 기준으로 계수하고 Python 객체 오버헤드는 벤치마크로 보정한다.
* MCP 종료 시 자동 소멸한다.

### B. 짧은 TTL 디스크 캐시 — 성능시험 통과 시 선택적 기본 활성화

```text
cache_mode = temporary_ttl
cache_ttl_hours = 24
cache_max_size_mb = 500
```

구현 원칙:

* 원문 ZIP 전체가 아니라 정규화된 파싱 텍스트만 저장한다.
* 접수번호를 파일키로 사용한다.
* gzip 압축은 낮은 압축수준을 사용하거나, 벤치마크상 더 빠르면 압축하지 않는다.
* 원자적 임시파일 저장 후 교체한다.
* 캐시 쓰기는 1개의 제한된 writer queue를 사용하고, 큐가 차면 검색을 지연시키지 않고 해당 캐시 쓰기를 생략한다.
* 만료 파일 정리는 매 요청마다 전체 스캔하지 않고, 서버 시작 시 또는 일정 확률의 lazy cleanup으로 처리한다.
* 캐시 읽기 오류나 손상은 해당 파일을 삭제하고 정상 다운로드로 폴백한다.

활성화 기준:

* 캐시 쓰기로 인한 전체 검색시간 증가가 **5% 이하**이고,
* 문서당 캐시 쓰기 p95가 **100ms 이하**이며,
* 유휴 메모리·디스크 사용량 목표를 해치지 않을 때만 기본 활성화한다.

위 기준을 충족하지 못하면 디스크 TTL 캐시는 기본 비활성화하고 세션 메모리 캐시만 사용한다.

### C. 회사 고유번호 캐시

`CORPCODE.xml`은 사용자 검색 코퍼스가 아닌 참조 데이터이므로 TTL 1일 디스크 캐시를 허용한다. 기존 코드의 회사명·종목코드 확인 로직은 재사용하되 재무정보 조회 기능은 제외한다.

## 3.3 경량 검색 감사 로그

법률 리서치의 재현성을 위해 기본적으로 검색 과정의 메타데이터를 JSONL 형식으로 기록한다.

기본 기록 항목:

* 검색 ID와 검색일시
* 사용자의 원 질의와 정규화된 질의 해시
* 검색 모드와 요청 결과 수
* 실제 검색기간, 공시유형 및 회사범위
* 사용한 검색어 변형
* 검색 채널과 폴백 여부
* 목록 API 호출 수, DART 본문검색 호출 수, 원문 다운로드 수
* 후보 접수번호와 최종 검증 접수번호
* 제외 사유 코드
* 정정본·철회본 처리 결과
* 소요시간, 캐시 적중 수, 재시도 수 및 오류코드
* 결과의 완전성 등급

기본적으로 기록하지 않는 항목:

* OpenDART API 키
* 공시 원문 전체
* 사용자의 다른 대화내용
* 불필요한 HTTP 헤더 또는 쿠키

성능 원칙:

* 요청별 상세 로그를 동기식으로 여러 번 쓰지 않고 검색 종료 시 한 번에 요약 기록한다.
* 임시파일을 거쳐 원자적으로 저장한다.
* 로그 기록 실패가 검색 실패로 이어지지 않도록 한다.
* 기본 보존기간은 30일, 최대 크기는 50MB로 하고 초과 시 오래된 로그부터 삭제한다.
* 사용자가 원하면 `audit_log = off`로 비활성화할 수 있다.

## 3.4 영구 저장과 인덱스는 사용자 승인 후 실행

다음 작업은 반드시 사전 승인을 받는다.

* 특정 회사의 전체 공시 인덱싱
* 장기간 시장 전체 공시 수집
* 원문 ZIP 또는 파싱 본문의 장기 보관
* 정기 신규 공시 증분수집
* SQLite FTS 또는 벡터 DB 구축

실행 전 다음 내용을 제시한다.

```text
수집 대상과 기간
검색할 공시유형
예상 API 및 본문검색 요청 수
예상 후보·다운로드 문서 수
예상 저장용량
예상 실행단계와 중단·재개 방법
결과 및 캐시 삭제 방법
```

승인 계획은 짧은 유효기간의 `plan_id`로 관리한다. 서버 재시작 또는 유효기간 경과 시 다시 미리보기를 수행한다. 승인되지 않은 `build_persistent_index` 호출은 실행하지 않는다.


## 4.2 병렬 처리 권장값

외부 서비스에 과도한 부하를 주지 않으면서 원문 최대 40건을 처리하기 위한 초기 권장값은 다음과 같다.

```text
DART 본문검색: 순차 1개
OpenDART 목록 기간창: 최대 2개
OpenDART 원문 다운로드: 기본 3개
원문 다운로드 재시도: 최대 2개
로컬 XML 파싱·키워드 검색: 최대 4개
TTL 캐시 writer: 1개
```

적응형 감속:

* HTTP 429, 반복 타임아웃 또는 5xx가 발생하면 원문 동시성을 3→2→1로 낮춘다. OpenDART `020`은 요청 제한 초과이므로 감속·백오프하지 않고 즉시 검색을 중단하고 요청예산 안내를 반환한다.
* 정상 요청이 일정 횟수 누적되어도 한 검색 중 자동으로 4개 이상까지 올리지 않는다.
* 배치 모드에서만 성능시험 결과가 양호한 경우 원문 다운로드 최대 4개를 허용한다.
* 캐시 적중 문서의 로컬 파싱은 외부 요청이 아니므로 최대 4개를 유지할 수 있다.

기존 코드의 목록 3개·원문 4개 병렬 구조는 배치모드의 상한 참고값으로만 활용하고, 대화형 MCP 기본값은 위와 같이 더 보수적으로 설정한다.

## 4.3 검색 호출 예산

### 빠른 검색

```text
OpenDART 목록 요청: 최대 10회
DART 본문검색 요청: 최대 3회
원문 다운로드: 최대 10건
최종 결과: 최대 10건
외부요청 동시성: 목록 2 / 원문 3
소프트 시간예산: 25초
하드 시간예산: 45초
```

### 표준 검색 — 기본값

```text
OpenDART 목록 요청: 최대 40회
DART 본문검색 요청: 최대 10회
원문 다운로드: 최대 40건
최종 확인 결과: 최대 20건
예비 후보: 최대 10건
외부요청 동시성: 목록 2 / 원문 3
소프트 시간예산: 60초
하드 시간예산: 90초
```

* 소프트 시간예산에 도달하면 확보된 결과가 충분한지 평가하고 조기 종료할 수 있다.
* 하드 시간예산 또는 호출예산에 도달하면 부분 결과와 continuation token 또는 배치 권고를 반환한다.
* 사용자가 20건 미만을 요청하면 요청 수에 맞춰 다운로드 예산도 축소한다.
* DART 본문검색에서 충분한 후보가 확보되면 40건을 모두 다운로드하지 않는다.

DART 본문검색 요청예산은 HTML 결과행 예산과 분리하지 않는다. 새 질의 1개는 보수적으로 모드설정 POST 1회와 첫 결과 POST 1회를 사용하고, 같은 쿠키 세션·같은 질의의 추가 페이지는 `/search.ax` 1회씩 사용한다. 따라서 표준예산 10회는 상태진단을 제외해도 새 질의 하나에서 최대 9개 결과페이지, 단계 0.5 실효값으로 약 90행에 해당하며 중복제거 후 고유 접수번호는 그보다 적을 수 있다. 여러 변형어를 사용하면 변형마다 보수적으로 모드설정 비용을 다시 합산한다.

```text
result_page_budget = dart_request_budget - health_check_requests - mode_setup_requests
fully_pageable = ceil(search_count / 10) <= result_page_budget
```

`fully_pageable=true`이면 기간·질의 범위 안에서 마지막 페이지까지 순회할 수 있다. false이면 정밀 질의로 축소하거나, 단계 0.6에서 검증된 경우 날짜창 분할로 완주 단위를 줄인 뒤 다시 계산한다. `상계납입` 42건은 5페이지로 표준예산 안에서 완주할 수 있지만 `출자전환` 13,084건과 `상계 납입` 50,864건은 대화형 전수조회 대상이 아니다. 대표 사례면 상위 고유 접수번호를 검증하고, 전수성이 필요하면 배치 승인으로 전환한다.

### 심층·배치 검색

자동으로 시작하지 않는다. 표준 검색 전후에 예상 범위를 계산하여 사용자 승인을 요청한다.

## 4.4 결과물 크기 제한

기본 응답:

* 최종 확인 사례 최대 20건
* 예비 후보 최대 10건
* 공시당 핵심 근거 1~3개
* 공시당 발췌 총길이 제한
* 회사명, 공시명, 접수일, 접수번호, 원문 링크
* 정정·철회·최종 유효본 상태
* 검색어 매칭 사실과 법률적 판단 분리
* 검색범위, 호출량, 캐시, 폴백 및 완전성 정보

동일 거래의 원공시와 여러 정정공시가 있으면 기본 결과에서는 **최종 유효본 1건을 사건 단위로 표시**하고, 전체 정정 체인은 `get_disclosure_evidence`에서 제공한다. 사건 식별은 원문에 기재된 원접수번호·정정대상 접수번호·정정사항표를 우선 사용한다. 회사·정규화 보고서명·접수일 근접성은 보조수단이며, 명시적 연결 근거가 없으면 서로 합치지 않고 `chain_confidence=uncertain`으로 표시한다. 이로써 결과 20건이 같은 사건의 버전들로 채워지는 문제와 독립 사건이 잘못 합쳐지는 문제를 함께 방지한다.

## 4.5 continuation 방식

대화형 검색이 예산을 초과하면 부분 결과를 반환한다.

```json
{
  "status": "partial",
  "confirmed_results": 12,
  "coverage": {
    "searched_from": "2026-07-16",
    "searched_to": "2021-01-01"
  },
  "continuation_token": "cursor_xxx",
  "batch_research_recommended": false,
  "reason": "standard_search_budget_reached"
}
```

토큰이 커지지 않도록 기본적으로 UUID가 짧은 TTL 세션상태를 참조하게 한다. 세션상태에는 현재 기간, 페이지, 처리한 접수번호 해시 및 검색어 변형을 저장한다. 서버 재시작으로 상태가 사라지면 사용자에게 재검색이 필요함을 명시한다.

휴대 가능한 토큰이 필요한 경우에만 작은 base64 JSON 커서를 사용한다. 토큰은 로컬 단일 사용자 환경에서 암호화·서명하지 않되, 입력값 검증과 최대 길이 제한을 적용한다.

## 4.6 DART 어댑터 회로차단기

회로차단기는 배치검색 전체를 15분 뒤 중단시키는 장치가 아니다. **고장 난 DART 본문검색 채널을 일정 시간 재호출하지 않고 OpenDART 폴백으로 즉시 우회하는 보호장치**다. 동일한 오류가 검색어 변형마다 반복되면 각 요청이 타임아웃될 때까지 기다리게 되고, DART에도 불필요한 요청이 누적되므로 필요성이 있다.

다만 모든 장애에 15분을 일률 적용하면 일시적인 통신장애에도 복구 확인이 늦어질 수 있으므로 장애유형에 따라 다음처럼 적용한다.

### A. 구조·접근 장애 — 15분 차단

다음은 단시간 내 자연복구 가능성이 낮으므로 **15분간 open**한다.

* 연속 2회 HTML 구조 파싱 실패
* 필수 검색결과 요소·필드 소실
* 로그인·보안·차단 페이지 반복 반환
* 요청 파라미터 또는 세션 방식 변경이 의심되는 응답
* 결과 행이 없고 단계 0에서 고정한 명시적 “검색결과 없음” 마커도 발견되지 않는 응답이 상태진단 재시도에서도 연속 2회 반환됨

정상 0건 페이지는 장애로 보지 않는다. 단계 0에서 DART 결과 페이지의 명시적 0건 마커 `조회 결과가 없습니다.`와 `검색건수: 0`을 golden fixture로 고정한다. 마커는 결과 컨테이너 내부에서 정규화해 판정하고, 페이지 다른 영역의 동일 문구는 사용하지 않는다. 결과 행 또는 정상 0건 마커 중 하나가 확인되면 채널은 정상이다. 둘 다 없으면 첫 응답은 `structure_failure_candidate`로만 기록하고, 저비용 상태진단에서도 반복될 때 구조장애로 승격한다. 합성 fixture는 분기 테스트에만 사용하며 실제 장애 발생률의 근거로 세지 않는다.

15분은 구조 변경을 해결하는 시간이 아니라, 같은 세션에서 실패를 반복하지 않기 위한 냉각시간이다. 이 경우 사용자에게 구조 변경·접근정책 확인 필요성을 명확히 알린다.

### B. 일시적 네트워크 장애 — 3분 차단

다음은 비교적 빨리 복구될 수 있으므로 연속 3회 실패 후 **3분간 open**한다.

* 연결 타임아웃
* 일시적 5xx
* 연결 리셋 또는 DNS 오류

3분 후 half-open 상태확인 1회를 수행하고, 성공하면 즉시 정상화한다. 동일 장애가 다시 발생하면 두 번째부터 5분, 세 번째부터 최대 15분으로 단계적으로 늘릴 수 있다.

### 공통 동작

* open 상태에서는 DART 본문검색을 재시도하지 않고 즉시 OpenDART 폴백
* 차단시간 종료 후 1회의 half-open 상태확인 요청
* 성공 시 복구, 실패 시 장애유형에 맞는 차단시간 재적용
* 상태는 세션 내 또는 짧은 TTL 상태파일로만 유지
* 배치 리서치 중 회로차단기가 열리면 먼저 체크포인트를 저장한다.
* 승인된 실행구간 안에서 OpenDART 폴백을 계속할 수 있으나, 예상 원문 수 또는 예상시간이 미리보기보다 25% 이상 증가하면 **사용자가 선택한 현재 실행구간 종료 시** 추가 승인을 요청한다.

회로차단 상태는 사용자에게 숨기지 않는다.

```json
{
  "code": "DART_FULLTEXT_CIRCUIT_OPEN",
  "failure_class": "structure_or_access",
  "message": "DART 본문검색 어댑터의 구조 또는 접근 방식 변경이 의심되어 15분간 해당 채널을 재호출하지 않습니다. 현재 검색은 OpenDART 원문검색으로 대체합니다.",
  "blocked_minutes": 15,
  "blocked_until": "실제 차단 종료시각",
  "fallback_source": "opendart_document_search"
}
```

`blocked_until`, 장애유형, half-open 결과는 검색 감사 로그에도 기록한다.

## 4.7 사용자 승인형 배치 리서치

배치 권고는 자주 표시되지 않도록 **원시 공시 수나 단순 기간 길이만으로 판단하지 않는다.** DART 본문검색, 공시유형, 보고서명 및 회사관계 필터를 적용한 뒤의 예상 원문 수와 예상 소요시간을 기준으로 한다.

일반 대화형 검색에서 이 예상값은 전체 창 사전조사로 만들지 않는다. 이미 처리한 첫 활성 창의 `total_count`를 남은 기간에 외삽하고, 경계 판정이 불확실할 때만 최대 2개 창을 추가 표본조회한다. 배치 미리보기에서는 모든 창의 첫 페이지를 조회해 실측 기반으로 다시 계산한다. 응답에는 `estimation_basis`, `sampled_window_count`, `confidence`를 표시한다.

DART 본문검색이 포함되면 응답의 `검색건수`를 OpenDART `total_count`와 별도 신호로 저장한다. 대표 사례 검색에서는 상위 후보 조기 종료 가능성을 함께 고려하고, 전수성 검색에서는 `ceil(search_count / effective_page_size)`를 페이지 상한의 1차 추정값으로 사용한다. 단계 0.5에서 `effective_page_size=10`을 관찰했지만 같은 접수번호가 반복될 수 있으므로 `dedupe_yield_rate`와 `estimated_unique_candidates`도 함께 계산한다. `search_count`가 수천 건이어도 목표가 대표 사례 10건이면 즉시 배치를 강제하지 않지만, 정밀화 후 필요한 후보 페이지와 검증 통과율을 반영한 예상 원문 수가 예산을 넘으면 배치를 권고한다.

전수성 배치에서 단일 질의의 예상 결과페이지 수가 페이지 예산을 넘으면 `b_startDate/b_endDate`로 질의를 완주 가능한 기간 구간들로 나눠 실행한다. 단계 0.6에서 `상계납입` 전체기간의 고유 접수번호 9건과 비중첩 세 날짜창의 합집합 9건이 일치했고 시작일·종료일은 포함 경계로 확인됐다. 날짜창은 `[start, end]`를 사용하고 다음 창은 `end+1일`에서 시작하며, 창별 완료·전체 기간 연속성·전역 `rcept_no` 중복제거를 필수 검증한다.

DART 요청 예상시간의 하한은 초기 안전값인 요청 시작간격 1,000ms에서 계산한다. 새 쿠키 세션 또는 새 검색 모드의 첫 페이지는 검색모드 설정 POST와 결과 POST 2회로 계산한다. 같은 세션·같은 검색 모드에서는 검색어가 바뀌어도 결과 POST 1회만 합산하며 추가 페이지도 결과 POST 1회다. 단계 0.6의 3개 검색어 전환에서 직접 결과와 모드 재설정 대조군이 정확히 일치했다. 세션 상태진단 요청은 별도 합산한다.

```text
dart_rate_floor_seconds = max(0, estimated_dart_requests - 1) * 1.0
estimated_network_floor_seconds = dart_rate_floor_seconds + opendart_rate_floor_seconds
```

실제 예상시간은 네트워크 p50/p95와 원문 검증시간을 더해 계산하며, rate 하한보다 작게 표시하지 않는다.

### 배치를 권고하지 않는 경우

다음 경우에는 기간이 길거나 초기 후보가 많아도 대화형 검색을 먼저 수행한다.

* 필터 후 예상 원문 확인 수가 40건 이하임
* 원문 확인 수가 40건을 넘더라도 조기 종료로 요청 결과 수를 확보할 가능성이 높음
* 사용자가 5~20건의 대표 사례만 요청했고, DART 본문검색으로 우선순위 후보가 충분히 확보됨
* 특정 회사 검색처럼 `corp_code`로 범위가 좁혀짐

### 선택적 배치 안내

다음 경우에는 대화형 결과를 먼저 반환한 뒤, 확인창을 강제하지 않는 선택 안내만 1회 표시할 수 있다.

* 필터 후 예상 원문 수가 41~80건이고 대화형 검색으로 일부 결과는 확보함
* 한 번의 continuation이 필요하지만 목표 건수 달성 가능성이 여전히 높음
* 완전성보다 대표 사례 확보가 목적인 장기간 검색

### 배치 승인 요청

다음 경우에만 `preview_batch_research`를 권고하고 사용자 확인을 요청한다.

* 사용자가 “전수”, “빠짐없이”, “가능한 모든 사례” 등 완전성 높은 조사를 명시적으로 요구함
* 필터 후 예상 원문 확인 수가 80건을 초과하고, 조기 종료로 목표를 달성할 가능성이 낮음
* 예상 실행시간이 약 5분을 초과함
* 대화형 continuation을 2회 수행했는데도 목표 건수를 확보하지 못함
* DART 본문검색 폴백으로 예상 원문 수 또는 예상시간이 최초 계획보다 25% 이상 증가함
* CSV·JSON 저장, 체크포인트 또는 장시간 배치 출력이 명시적으로 필요함

배치 권고는 동일한 `search_lineage_id`에서 한 번만 표시한다. 사용자가 거절하면 해당 검색계보에서는 세션 종료 또는 30분 동안 다시 권고하지 않는다. 다만 사용자가 검색범위를 확대하거나, 예상 원문 수·시간이 기존보다 50% 이상 증가한 경우에는 다시 안내할 수 있다.

자동 실행하지 않고 다음 순서를 따른다.

```text
표준검색 또는 사전범위 계산
→ batch_research_recommended 반환
→ preview_batch_research 호출
→ 범위·예상요청·문서수·저장정책 제시
→ 사용자 명시적 승인
→ run_batch_research(plan_id) 실행
```

미리보기 단계에서는 배치 실행 여부뿐 아니라 **추가 승인을 받을 실행 간격**도 사용자에게 묻는다.

권장 선택지는 다음과 같다.

```text
5분   매우 세밀한 통제, 확인 빈도 높음
10분  기본 권고값, 통제와 편의의 균형
15분  중간 규모 조사에 적합
30분  넓은 범위 조사, 확인 빈도 낮음
```

클라이언트가 선택형 입력을 지원하지 않으면 사용자가 분 단위 숫자를 답하도록 한다. 허용범위는 5~30분이며, 응답이 없거나 유효하지 않으면 10분을 기본값으로 제시하되 자동 실행하지 않고 다시 승인을 받는다.

* 기간창·페이지별 체크포인트를 남겨 중단 후 재개할 수 있게 한다.
* 기본 원문 동시성은 3, 벤치마크가 양호한 경우에만 4까지 허용한다.

* 배치 결과와 체크포인트의 보존기간·삭제방법을 미리보기에서 표시한다.
* 영구 원문보관이나 정기 증분수집은 별도 승인이 필요하다.
* 배치 최초 승인에는 `confirmation_interval_minutes`가 반드시 포함되어야 한다.
* 한 번의 실행구간은 사용자가 선택한 5·10·15·30분을 따른다. 내부적으로는 선택시간 30초 전을 소프트 제한, 선택시간을 하드 제한으로 사용한다.
* 선택한 구간이 끝났는데 작업이 남아 있으면 체크포인트를 저장하고 `continuation_confirmation_required` 상태를 반환한다.
* 사용자가 추가 진행을 승인해야 동일한 시간의 다음 구간을 실행한다. 사용자는 후속 승인 시 실행간격을 변경할 수도 있다.
* 클라이언트가 MCP elicitation을 지원하지 않아도 별도 후속 도구 호출 방식으로 동일하게 동작한다.

## 4.8 배치와 대화형 검색의 공통 엔진

두 모드가 서로 다른 검색 논리를 갖지 않도록 다음 공통 컴포넌트를 사용한다.

```text
기간분할
OpenDART 오류 처리·백오프
회사 식별
공시유형·보고서명 필터
원문 다운로드·인코딩 파싱
검색어 확장
정정본·동일 사건 묶기
정정 전·후 구조적 diff 및 변경방향 판정
근거 발췌
감사 로그 스키마
```

대화형 모드는 예산과 응답크기를 제한하고, 배치 모드는 사용자 승인 후 더 넓은 범위와 파일 출력을 허용하는 차이만 둔다.

# 5. MCP 장시간 작업 기능에 관한 판단

MCP Tasks는 장시간 실행을 위한 유용한 방향이지만 공식 Tasks extension은 여전히 실험적 상태다. 따라서 초기 버전은 특정 Tasks lifecycle이나 미래 사양을 전제로 구현하지 않는다.

초기 원칙:

* 일반 검색은 제한된 한 번의 도구 호출로 처리한다.
* 검색예산 초과 시 continuation 또는 배치 권고를 반환한다.
* 배치는 명시적 승인 후 별도 실행기로 수행한다.
* 진행률·취소 기능은 현재 사용 중인 MCP 클라이언트가 지원하는 범위에서만 선택적으로 제공한다.
* Tasks extension이 안정화되고 주요 클라이언트 지원이 확인된 뒤 배치 실행의 선택적 전송계층으로 검토한다.

# 6. 주요 질의별 실행 설계

## 6.1 특정 회사의 특정 주제 공시

예:

> SK하이닉스의 ADS와 원주 전환 관련 공시를 찾아줘.

절차:

1. 회사명·종목코드로 `corp_code` 확인
2. 회사 기준 OpenDART 목록 검색
3. ADS, ADR, 예탁증권, 예탁주식, 원주, 기초주식 등 표현 확장
4. DART 본문검색 병행
5. 후보 최대 40건 내 원문 확인
6. 정정공시와 최종 유효본 정리
7. 관련성이 높은 사례 최대 20건 반환

## 6.2 공개매수 이후 포괄적 주식교환

예:

> 특정 회사에 대하여 공개매수 관련 공시가 이루어진 이후 포괄적 주식교환 관련 공시가 이루어진 사례를 10건 찾아줘.

절차:

1. DART 본문검색 및 OpenDART 상세유형으로 공개매수 후보 수집
2. 공개매수자와 대상회사 식별
3. 대상회사 또는 관계회사 기준 후속 주식교환·이전 공시 검색
4. 공개매수 공시일과 주식교환 공시일의 선후관계 확인
5. 공개매수자·완전모회사, 대상회사·완전자회사 관계 검증
6. 정정·철회 및 거래 무산 여부 확인
7. 동일 거래를 사건 단위로 묶어 최대 20건 반환

대상회사 식별 우선순위:

1. 보고서명 및 목록 필드 파싱
2. 대상회사의 공개매수 의견표명 등 역방향 공시
3. 실패한 건만 원문 다운로드

표준예산으로 10건을 확보하지 못하면 부분 결과와 배치 권고 여부를 함께 반환한다.

## 6.3 유상증자 중 상계납입

예:

> 유상증자 공시들 중 상계납입을 한 사례들을 찾아줘.

이 절은 단계 0.5 `GATE-FLAGSHIP-TERMS`를 통과한 플래그십 S3 사용사례다. 실측한 1년 창에서 `상계납입` 42건과 `주금납입채무와 상계` 54건을 확인했고, 원문 6건을 직접 상계납입 4건·출자전환 2건으로 검증했다. 이 통과는 해당 질의의 기술적 사용과 우선순위를 확정한 것이며 전체 시장 recall이나 모든 기간의 검색건수를 보장하지 않으므로 OpenDART 원문검색 폴백은 유지한다.

절차:

1. DART 본문검색에서 `상계납입`과 `주금납입채무와 상계` 정밀 질의를 먼저 실행
2. 유상증자·유무상증자·제3자배정 등 보고서명 필터 적용
3. 수량이 부족할 때 `출자전환`과 거래문맥 변형을 추가하고, broad가 필요할 때만 띄어쓰기 변형 실행
4. 모든 질의·페이지의 후보를 접수번호로 전역 중복제거
5. OpenDART 메타데이터로 회사·공시일·정정 여부 확인
6. 후보 최대 40건 원문 확인
7. 직접 상계납입, 현물출자, 출자전환 및 단순 회계상 상계를 구별
8. 기계적 매칭과 법률적 분류를 분리하여 최대 20건 반환

검색어 예시:

```text
상계납입
주금납입채무와 상계
출자전환

# broad 확장 — 정밀 질의가 부족하거나 사용자가 명시한 경우만
상계 납입
주금 납입 채무와 상계
납입대금과 채권을 상계
대여금채권과 상계
채권의 출자전환
현금 납입 없이
```

`채권의 출자전환`은 `출자전환` 대비 첫 페이지 신규 접수번호 기여가 0이었으므로 기본 자동확장에 포함하지 않는다. 띄어쓴 `상계 납입`은 토큰이 떨어진 단순 납입·회계상 상계 문서까지 대량 매칭하므로 검색건수와 원문 통과율을 확인하면서 broad 단계에서만 사용한다.

결과 표현:

```text
지정 검색어 변형과 원문 확인으로 발견된 사례
```

전수검색이 아닌 경우 “시장 전체의 모든 사례”라고 표현하지 않는다.

## 6.4 정정된 내용 자체를 기준으로 한 사례 검색

예:

> 납입일이 연기되도록 정정공시된 유상증자 사례를 찾아줘.

> 매매대금이 감액되는 내용으로 정정된 공시 사례를 찾아줘.

> 전환가액 산정방식이 변경되도록 정정공시된 사례를 찾아줘.

수행 가능하다. 다만 정정공시의 최종 본문에 검색어가 존재하는지만 확인해서는 부족하고, **정정 전과 정정 후의 차이 자체**가 사용자 조건에 해당하는지 검증해야 한다.

<a id="AMEND-CHAIN-001"></a>
검색 절차:

1. 질의에서 정정 대상 개념, 변경 방향(추가·삭제·증가·감소·연기 등), 공시유형과 기간, 자진정정인지 감독당국 조치인지 여부를 추출한다.
2. OpenDART 목록 단계에서 `rm_flags`와 `report_name_prefixes[]`를 먼저 해석한다. `rm`에 `정`이 없는 최신 후보는 S7에서 잠정 최종본으로, `철`이 있으면 철회후보로 분류하되 두 값 모두 사건 연결이나 최종 유효성의 확정 근거로 사용하지 않는다.
3. DART 본문검색에는 공식 접두어와 대상 문구를 조합한다. 공식 접두어는 `[기재정정]`, `[첨부정정]`, `[첨부추가]`, `[변경등록]`, `[연장결정]`, `[발행조건확정]`, `[정정명령부과]`, `[정정제출요구]`다. 일반적인 `[정정]` 접두어를 공식값으로 가정하지 않는다.
4. 본문 변경 검색의 기본 대상은 `[기재정정]`과 거래유형상 본문 변경을 수반하는 `[변경등록]`·`[연장결정]`이다. `[첨부정정]`·`[첨부추가]`는 기본적으로 본문 diff 대상에서 제외하고 첨부변경 검색일 때만 포함한다.
5. `[발행조건확정]`은 정정과 별도 유형으로 저장한다. 다만 전환가액·발행가액 등 조건변경 또는 확정 사례를 찾는 질의에서는 `amendment_rules.yaml`의 `include_final_terms=true` 규칙에 따라 포함할 수 있다.
6. `[정정명령부과]`와 `[정정제출요구]`는 회사의 자진정정과 구분하여 `amendment_origin=regulator_required`로 분류하고 필터 가능하게 한다.
7. OpenDART `last_reprt_at`은 단계 0의 3개 사건에서 N이 전체 체인, Y가 해당 사건 최종 접수를 반환했다. 다만 Y 응답에도 동일 회사·기간·정규화 보고서명의 독립 사건이 존재할 수 있으므로 Y는 후보 축소에만 사용하고 사건 식별키나 최종본 증명으로 사용하지 않는다.
8. 정정공시 원문의 앞부분 정정사항표·정오표, 원 접수번호, 정정대상 보고서 정보를 1차 체인 연결수단으로 사용한다. 보고서명·회사·시점 매칭은 보조수단이며, 보조수단만으로 연결한 체인은 `chain_confidence=uncertain`으로 표시한다.
9. 정정공시에 명시된 `정정사항`, `정정 전`, `정정 후`, `정정사유` 표를 우선 추출한다.
10. 명시적 정정표가 없거나 파싱이 불완전하면 동일 섹션·표·행 기준의 구조적 diff를 수행하고, 마지막 수단으로 정규화된 본문 diff를 사용한다.
11. 사용자가 요청한 변경내용이 실제 diff에 존재하는 경우만 최종 사례로 확정한다.
12. 여러 차례 정정된 사건은 원공시→각 중간본→최종본의 단계별 delta와 원공시→최종본 누적 delta를 모두 보존한다.

결과 필드:

```json
{
  "corp_name": "예시회사",
  "base_report_name": "유상증자결정",
  "original_rcept_no": "...",
  "amended_rcept_no": "...",
  "amendment_date": "2026-01-15",
  "changed_section": "납입일",
  "change_type": "date_postponed",
  "before_excerpt": "납입일: 2026년 1월 20일",
  "after_excerpt": "납입일: 2026년 2월 20일",
  "amendment_reason": "관계기관 일정 협의",
  "mechanical_match": true,
  "legal_assessment": "납입일 연기 정정에 해당",
  "confidence": "high"
}
```

정정 전·후 비교는 일반 검색보다 문서 소모가 크므로 별도 예산을 적용한다.

```text
일반 대화형 검색 원문 한도: 최대 40건
정정비교 대화형 검색 원문 기본한도: 80건
정정비교 대화형 검색 절대한도: 120건
최종 결과 한도: 최대 20개 사건
```

80건은 정정 전·후 및 중간 정정본을 합산한 기본 문서 수다. 3~5개 버전의 정정 체인과 후보 탈락을 고려하면 80건으로 실제 확정 가능한 사건은 대략 8~12건일 수 있다. 사전추정상 20건 목표 달성이 어렵고 대화형 시간예산 안에 처리 가능할 때만 최대 120건까지 확장하며, 그렇지 않으면 배치 미리보기를 반환한다.

정정비교에서도 80건을 무조건 모두 내려받지 않는다. 명시적 정정표가 있는 공시, 검색어가 정정 전·후 열에 직접 나타나는 공시, 최신 유효본과 연결이 명확한 사건을 우선 처리하고 목표 결과를 확보하면 조기 종료한다. 80건 또는 정정비교 전용 시간예산에 도달하면 부분 결과와 continuation을 반환한다.

필터 후 예상 비교문서가 120건을 초과하고 조기 종료 가능성이 낮거나, 예상시간이 사용자가 선택할 수 있는 최소 배치구간인 5분을 넘는 경우에만 배치 미리보기를 권고한다.

# 7. 검색 범위와 완전성 표시

## 7.1 탐색 검색

* 빠른 사례 발견 목적
* 일부 검색어와 제한된 후보 원문 확인
* 완전성 보장 없음

## 7.2 범위 한정 검색

* 특정 회사, 기간, 공시유형 또는 지정 검색어 변형을 체계적으로 확인
* 실제 검색 범위를 명시
* DART 본문검색이 정상 작동하고 후보 원문이 충분히 검증된 경우에만 해당 등급 부여

## 7.3 배치·전수 지향 검색

* 장기간 또는 시장 전체를 기간 분할하여 조사
* 사용자 승인 후 실행
* 검색종료 조건, 미처리 문서 및 실패 문서를 명시
* 영구 인덱스가 없는 경우에도 “전수 완료”는 성공적으로 처리된 범위에 한해서만 주장

모든 결과에 포함할 진단정보:

* 실제 검색기간과 회사범위
* 검색한 공시유형
* 검색어 변형 목록
* DART 본문검색 정상·폴백 여부
* 후보 수, 원문 검증 수, 캐시 적중 수
* 제외·실패 수 및 주요 사유
* 조기 종료·부분 결과·배치 여부
* DART 본문검색 조기 종료 시 최신순 편향 여부
* 결과의 완전성 등급

# 8. MCP 도구 설계

초기 권장 도구:

## `search_disclosure_cases`

대화형 공시 사례검색을 수행한다.

```json
{
  "query": "공개매수 후 포괄적 주식교환 사례",
  "target_count": 10,
  "mode": "standard",
  "date_from": "2015-01-01",
  "date_to": "2026-07-16",
  "max_documents": 40,
  "cache_mode": "auto",
  "continuation_token": null
}
```

서버는 `max_documents`를 40 이하로 제한하고, `target_count`는 20 이하로 제한한다.
`query`가 정정 전·후 변경내용을 찾는 요청이면 내부적으로 `search_intent="amendment_diff"`를 적용한다. 별도 고수준 도구를 늘리지 않고 동일 도구가 후보수집·정정체인 연결·diff 검증을 오케스트레이션한다.

## `get_disclosure_evidence`

특정 접수번호의 원문 근거, 정정 체인 및 상세 발췌를 반환한다.

## `preview_batch_research`

대화형 예산을 넘는 검색의 범위·요청 수·문서 수·보존정책을 계산하며 실제 검색은 시작하지 않는다.

## `run_batch_research`

유효한 `plan_id`, 사용자의 명시적 승인 및 사용자가 선택한 실행구간(5·10·15·30분) 이후에만 최초 배치 리서치 구간을 실행한다.

## `continue_batch_research`

직전 사용자 선택 실행구간이 미완료로 끝난 경우, 저장된 체크포인트와 사용자의 추가 승인을 확인한 뒤 다음 선택 실행구간을 실행한다.

## `preview_persistent_index`

영구 인덱스 구축의 범위와 저장량을 계산한다.

## `build_persistent_index`

별도로 승인된 `plan_id`가 있을 때만 실행한다.

## `get_search_record` 또는 `export_search_record`

검색 ID 기준으로 감사 로그 요약을 확인하거나 JSON으로 내보내는 선택 도구다. 원문 전체는 포함하지 않는다.

### 도구 수 관리

호스트 LLM이 혼동하지 않도록 도구 설명에 다음을 명확히 한다.

* 일반 질문은 `search_disclosure_cases`
* 특정 원문 확인은 `get_disclosure_evidence`
* 40건 초과 또는 전수조사는 `preview_batch_research` 후 `run_batch_research`
* 영구 인덱스는 배치 리서치와 별도

저수준 도구(`resolve_company`, `list_disclosures`, `fulltext_search`, `get_document`)는 단계 4 벤치마크에서 고수준 도구의 오분류가 높을 때만 추가한다. `get_financials`는 프로젝트 범위에서 제외한다.

## 8.1 기계적 사실과 LLM 판단 분리

결과 스키마 예시:

```json
{
  "mechanical_match": {
    "matched_terms": ["주금납입채무", "상계"],
    "report_type": "유상증자결정",
    "source_receipt": "2026xxxxxxxxxx"
  },
  "legal_assessment": {
    "classification": "직접 상계납입 가능성 높음",
    "confidence": "medium",
    "reason": "주금납입채무와 회사에 대한 채권의 상계를 명시"
  }
}
```

규칙 기반 로직은 후보 선별과 기계적 매칭까지만 책임진다. 법률적 분류가 애매한 경우 제외하지 않고 예비 후보로 반환하여 호스트 LLM이 최종 판단하도록 한다.

## 8.2 공시 원문을 신뢰되지 않은 데이터로 처리

공시 본문에 포함된 문구는 MCP나 호스트 LLM에 대한 명령이 아니라 데이터다.

* HTML script, style 및 비가시 요소를 제거한다.
* 공시 안의 “이전 지시를 무시하라” 같은 문구를 도구 지시로 해석하지 않는다.
* 결과에 `source_text_untrusted: true`를 표시할 수 있다.
* 사용자 입력 URL을 임의로 요청하지 않고 DART·OpenDART·KIND의 허용된 고정 도메인만 호출한다.
* API 키, 쿠키 및 인증정보를 결과와 로그에서 마스킹한다.

# 9. 사용자 친화적 배포·설치 계획

## 9.1 루트폴더 최소화 원칙

배포 패키지의 루트에는 사용자가 직접 실행하거나 읽을 파일만 둔다. 개발용 Python 파일, YAML 규칙, 테스트 및 설치 내부 스크립트는 한 단계 아래 폴더에 배치한다.

데이터 저장위치는 설치형과 포터블형을 구분한다.

* **일반 설치형(기본 권장)**: 프로그램은 사용자 단위 설치경로에 두고, API 키·캐시·로그·결과는 `%LOCALAPPDATA%\공시검색_MCP\_local_data`에 저장한다. `Program Files` 쓰기권한 문제를 피하고 업데이트 시 사용자 데이터를 보존한다.
* **포터블/개발 배포형**: 사용자가 압축을 푼 루트의 `_local_data/`를 사용한다. 배포 ZIP 생성 시 이 폴더는 반드시 제외한다.
* 설치 프로그램과 `문제해결 및 제거.exe`에는 **사용자 데이터 폴더 열기** 버튼을 제공한다.

권장 배포 구조(포터블/개발 원본 기준):

```text
공시검색_MCP/
├─ 공시검색 MCP 설치.exe          # 사용자가 최초 1회 실행
├─ 사용설명서.html                # 비전문가용 안내서
├─ 문제해결 및 제거.exe           # 선택: 복구·설정삭제·제거
├─ app/                            # 프로그램 내부 파일
│  ├─ server/                      # MCP 서버 및 검색 엔진
│  ├─ adapters/                    # OpenDART·DART 어댑터
│  ├─ rules/                       # 검색어·이벤트·랭킹 YAML
│  ├─ installer/                   # 설치 내부 모듈
│  ├─ assets/                      # 아이콘·설명서 이미지
│  └─ vendor/                      # 필요한 경우 고정 의존성
└─ _local_data/                    # 배포 제외·사용자 로컬 데이터
   ├─ .env                         # DART_API_KEY
   ├─ cache/                       # TTL 캐시 및 corp_code 캐시
   ├─ logs/                        # 검색 감사 로그·설치 로그
   ├─ batch_results/               # 승인된 배치 결과
   ├─ checkpoints/                 # 배치 중단·재개 상태
   └─ temp/                        # 임시 다운로드·파싱 파일
```

`_local_data/`는 배포 ZIP, Git 커밋 및 업데이트 패키지에 포함하지 않는다. 개발 저장소에는 빈 폴더 유지용 파일조차 두지 않고, 설치 또는 최초 실행 시 자동 생성한다. 필요하다면 루트의 `.gitignore` 및 배포 빌드 스크립트에서 `_local_data/`를 이중으로 제외한다.

## 9.2 OpenDART API 키 저장과 설치 중 입력

API 키는 프로그램 코드나 Claude Desktop 설정에 직접 기록하지 않고, 실행형태에 따라 다음 파일에만 저장한다.

```text
일반 설치형: %LOCALAPPDATA%\공시검색_MCP\_local_data\.env
포터블형:    <프로그램 루트>\_local_data\.env
```

파일 내용:

```dotenv
DART_API_KEY=사용자가_입력한_API키
```

설치 절차:

1. 설치 프로그램이 `_local_data/`와 필요한 하위 폴더를 생성한다.
2. API 키 입력 화면에서 키를 붙여넣게 한다.
3. 입력값 앞뒤 공백을 제거하고, 비어 있거나 비정상적인 길이·문자가 있으면 저장 전에 안내한다.
4. 저비용 OpenDART 확인 요청을 1회 보내 유효성을 검사한다.
5. 검증 성공 시 `.env.tmp`에 먼저 기록한 뒤 원자적으로 `.env`로 교체한다.
6. 검증 실패 시 기존 `.env`가 있더라도 덮어쓰지 않고 다시 입력하거나 나중에 설정하도록 선택지를 제공한다.
7. 설치 로그와 화면에는 API 키 전체를 절대 출력하지 않고, 필요한 경우 앞 3자·뒤 2자만 마스킹하여 표시한다.

Windows에서는 가능하면 `_local_data/.env`의 ACL을 현재 사용자 계정 중심으로 제한한다. 다만 권한 설정 실패가 설치 전체를 막지는 않도록 경고 후 진행한다.

### 9.2.1 실행 중 API 키 문제 감지와 해결 안내

작동 중 다음 신호가 발생하면 단순 네트워크 오류와 API 키 문제를 구별한다.

* OpenDART 인증키 관련 상태코드 또는 “등록되지 않은 인증키”, “사용할 수 없는 인증키” 응답
* `.env` 없음, 빈 값, 잘못된 형식 또는 읽기권한 오류
* 설치 시에는 성공했으나 이후 키가 폐기·변경된 것으로 보이는 반복 인증 실패

API 키 문제로 판단되면 검색을 무한 재시도하지 않고 중단 또는 부분결과를 반환하며, 다음 해결절차를 사용자 언어로 안내한다.

1. `문제해결 및 제거.exe` 실행
2. **OpenDART API 키 확인/변경** 선택
3. 현재 키는 일부만 마스킹하여 표시
4. 새 키 입력 또는 OpenDART에서 키 재확인
5. 저비용 연결검사 수행
6. 성공 시 `.env`를 원자적으로 교체하고 검색을 처음부터 또는 체크포인트부터 재개

안내문에는 OpenDART 로그인 → 인증키 관리/발급 화면 → 키 복사 방법과, 기존 키가 일일 제한·중지·폐기 상태일 가능성을 함께 설명한다. API 키 전체는 화면·로그·감사기록에 절대 출력하지 않는다.

## 9.3 원클릭 설치파일

사용자는 루트의 `공시검색 MCP 설치.exe` 하나만 실행한다. 설치 프로그램은 다음을 순서대로 자동 처리한다.

1. 운영체제 및 권한 확인
2. Python 런타임 존재 여부와 버전 확인
3. 필요한 경우 앱 전용 가상환경 생성 또는 포함된 런타임 사용
4. 고정된 버전의 필수 패키지 설치
5. OpenDART API 키 입력 및 `_local_data/.env` 저장
6. Claude Desktop 설치·설정파일 경로 탐색
7. 기존 Claude 설정파일 백업
8. 기존 `mcpServers`를 보존하면서 본 MCP 항목만 안전하게 병합
9. MCP 서버 자체진단 및 OpenDART 연결검사
10. Claude Desktop 재시작 필요 안내
11. 설치결과와 문제해결 버튼 표시

설치 프로그램은 재실행 가능하고 멱등적이어야 한다. 이미 설치되어 있으면 `복구 설치`, `API 키 변경`, `Claude 설정 다시 등록`, `완전 제거` 중 하나를 선택할 수 있게 한다.

기존 `merge_config.py`에서 검증된 다음 방식을 재사용한다.

* 표준 설치와 MSIX 설치의 Claude 설정 경로 탐색
* 손상된 JSON을 덮어쓰지 않는 검증
* 기존 설정 백업
* 임시파일을 거친 원자적 저장
* 기존 MCP 서버 항목 보존

### 9.3.1 전용 런타임과 지원환경

초기 정식 지원범위는 다음으로 고정한다.

```text
운영체제: Windows 10/11 64비트
MCP 클라이언트: Claude Desktop 우선 지원
설치 방식: 관리자 권한이 필요 없는 사용자 단위 설치
런타임: 앱 전용 포함 Python 런타임 또는 완전 패키징된 실행파일
네트워크: DART/OpenDART HTTPS 접속 필요
```

사용자 PC의 시스템 Python과 전역 pip에 의존하지 않는다. 전용 런타임과 의존성 버전을 잠가 다른 프로젝트와 충돌하지 않게 한다. 다른 MCP 클라이언트는 초기에는 수동 설정 예시만 제공한다.

### 9.3.2 원자적 설치와 롤백

설치는 다음 단계가 모두 성공한 뒤에만 완료로 확정한다.

```text
설정·기존버전 백업 → 임시경로 설치 → .env 검증 → MCP 설정 병합
→ 서버 자체진단 → 성공 시 교체 확정 → 실패 시 전부 원상복구
```

API 키 검증 실패, 패키지 설치 실패, Claude JSON 병합 실패, 서버 기동 실패, 사용자 취소 및 동일 이름 MCP 충돌 시 롤백한다. 복구에 실패하면 백업 위치와 수동 복구 버튼을 보여준다.

## 9.4 사용설명서

루트에 `사용설명서.html`을 기본 제공한다. 브라우저에서 바로 열리고, 외부 프로그램 없이 읽을 수 있어야 한다. 필요하면 동일 내용을 PDF로도 생성하되 HTML을 기준본으로 한다.

필수 구성:

   * 인증키 신청 메뉴 이동
   * 이용 목적 입력 및 신청
   * 발급된 인증키 복사
   * 인증키를 다른 사람에게 공유하지 말아야 하는 이유
4. `공시검색 MCP 설치.exe` 실행 방법
5. 설치 중 API 키 입력 화면 설명
6. Claude Desktop 완전 종료·재시작 방법
7. 자연어 검색 예시
   * 특정 회사 공시 찾기
   * 공개매수 후 주식교환 사례 찾기

   * 특정 내용으로 정정된 사례 찾기

9. 배치 추가 승인 시간 선택 방법

## 9.5 설치파일 아이콘

설치파일 아이콘은 “공시 문서를 들여다보는 작은 돋보기 캐릭터” 또는 “문서 위에 앉은 귀여운 탐정 새”처럼 신뢰감과 친근함을 함께 주는 단순한 형태로 제작한다.

요건:

* 16·24·32·48·64·128·256px에서 식별 가능
* 작은 크기에서 글자나 세부 장식에 의존하지 않음
* 법률·금융 도구에 맞게 지나치게 장난스럽지 않음
* 투명 배경 ICO 포함
* 설치파일·제거파일·바로가기에서 동일한 시각체계 사용

아이콘 원본은 `app/assets/`에 저장하고, 중간 PNG·임시 ICO·빌드 캐시는 `_local_data/` 또는 빌드 전용 제외폴더에 둔다. 최종 배포에는 실행파일에 내장된 아이콘만 있어도 되며 원본 디자인 파일은 필수 배포대상이 아니다.

## 9.6 배포 자동화와 제외 규칙

루트의 `_local_data/` 외에도 개발 중 생성되는 다음 항목은 배포에서 제외한다.

```text
_local_data/
__pycache__/
.pytest_cache/
.venv/
build/
dist/
*.log
*.tmp
*.bak
.env
```

다만 사용자가 요청한 “배포 제외 파일을 한 폴더에 모은다”는 원칙에 따라, 런타임에서 생성되는 로그·캐시·결과·체크포인트·환경파일은 모두 `_local_data/` 아래로 강제한다. `build/`, `dist/`, 가상환경과 테스트 캐시는 개발 빌드 과정에서만 생기므로 가능하면 별도의 `developer/` 작업영역에서 생성하고 최종 배포 원본 폴더에는 남기지 않는다.

배포 스크립트는 allow-list 방식으로 다음만 포함한다.

```text
공시검색 MCP 설치.exe
사용설명서.html
문제해결 및 제거.exe
app/ 중 실행에 필요한 파일
```

즉 “제외할 파일을 빼는 방식”뿐 아니라 “허용한 파일만 담는 방식”을 병행하여 API 키나 사용자 결과가 실수로 배포되는 위험을 줄인다.

## 9.8 DART 본문검색 활성화 및 운영 게이트

DART 본문검색은 개인용 로컬 사용을 전제로 구현한다. 명시적 자동화 허용 근거와 허용 빈도는 단계 0에서 확인하지 못했지만, 사용자는 검색 성능이 나쁘지 않고 검색 품질 향상에 도움이 되는 경우 사용하기로 결정했다. 따라서 정책 미확인 자체를 차단 사유로 삼지 않고, 기술 상태와 품질 게이트를 통과하면 개인용 프로필에서 활성화한다. 다수 사용자 배포·서비스 제공·공유 서버 운영으로 범위가 넓어지면 이 결정은 무효이며 정책 검토를 다시 수행한다.

* 설치 또는 최초 기동 시 저비용 상태진단을 수행한다.
* 검색 실행 중 응답 구조·접근거부·세션 오류·반복 타임아웃을 감시한다.
* 동시성 1·요청 시작간격 최소 1,000ms로 시작한다.
* 고정 평가질의에서 OpenDART 단독 대비 후보 recall 또는 최종 검색 품질이 향상되고 지연이 허용범위이면 후보검색 채널로 사용한다.
* 품질 향상이 없거나 지연·오류가 기준을 넘으면 On-demand 채널로 강등한다.
* 장애나 접근거부가 감지되면 자동으로 해당 채널을 비활성화하고 OpenDART 원문검색으로 전환한다.
* 구조 또는 접근방식 변경이 의심되면 그 사실과 확인 필요성을 사용자에게 안내한다.
* 명시적 금지정책이 확인되면 제품 업데이트에서 해당 어댑터를 비활성화 또는 제거한다.
* `/dsaf001/main.do`는 사용자 링크 제공에만 사용하고 자동 본문 수집은 OpenDART 문서 API로 수행한다.

DART 본문검색이 없어도 핵심 기능이 작동하도록 OpenDART 폴백 경로를 독립적으로 테스트한다.

개인용 활성화 품질 게이트는 최소 20개의 고정 평가질의에서 OpenDART 단독 전략과 비교한다. 다음을 모두 만족하면 “성능이 나쁘지 않고 품질 향상에 도움”으로 판정한다.

* 접근거부·구조장애 0건, 정상 0건과 정상 결과를 모두 정확히 분류
* recall이 5%p 이상 향상되거나 희귀문구 질의의 20% 이상에서 추가로 검증된 관련 사건을 1건 이상 발견
* precision 하락이 2%p 이내
* 표준 검색 p95가 전체 하드 예산 90초 이내이며 OpenDART 단독 대비 25%를 넘게 악화되지 않음

기준을 충족하지 못하면 DART 본문검색을 제거하지는 않되 기본 전략에서 On-demand 보조채널로 강등한다.

## 9.9 검색 완료 후 결과파일 제공 여부 확인

대화형 검색 결과를 먼저 화면에 제시한 뒤, 파일 생성이 유용한 검색에서는 다음 선택을 묻는다.

```text
검색 결과를 파일로 저장해 드릴까요?
1) CSV  2) JSON  3) CSV+JSON  4) 파일 불필요
```

사용자가 이미 “파일로 줘”, “CSV로 정리해줘”라고 요청한 경우에는 다시 묻지 않고 해당 형식으로 생성한다. 결과가 0건이거나 매우 단순한 경우에는 파일 생성 질문을 생략할 수 있다. 기본값은 자동 저장하지 않음이다. 파일 생성 시에는 검색조건·기간·검색어·완전성 등급·폴백 여부를 메타데이터로 포함한다.

## 9.11 출력·입력 보안

### CSV 안전장치

외부 공시문구가 `=`, `+`, `-`, `@`, 탭 또는 캐리지리턴으로 시작하면 Excel 수식으로 실행되지 않도록 텍스트 이스케이프를 적용한다. UTF-8 BOM 또는 사용자 환경에 적합한 인코딩을 사용하되, 원문값과 안전처리 여부를 JSON에는 구분해 기록한다.

### ZIP·XML 안전장치

* ZIP 내부 경로를 파일시스템에 직접 해제하지 않고 메모리/안전 임시영역에서 처리
* ZIP 파일수, 단일 파일크기, 총 압축해제크기 및 압축률 제한
* XML 외부 엔터티와 외부 리소스 로딩 비활성화
* 단일 공시 최대 텍스트 크기 제한 및 초과 시 부분추출 경고
* 비표준 인코딩·손상파일은 실패사유를 기록하고 다른 문서 처리를 계속

## 9.12 배포 신뢰성·업데이트·자체진단

* 실행파일에 제품명·버전·게시자·빌드일 정보를 포함한다.
* 가능하면 코드서명하고, 미서명 개인용 빌드에는 Windows SmartScreen 경고 가능성을 사용설명서에 안내한다.
* 배포 ZIP과 설치파일의 SHA-256 체크섬을 생성한다.
* 의존성 버전을 잠그고 빌드 재현성을 테스트한다.
* 업데이트는 `app/`, 설명서, 설치·복구도구만 교체하고 사용자 데이터는 보존한다.
* 업데이트 전 기존 버전과 설정을 백업하고 새 버전 자체진단 실패 시 롤백한다.

기술 traceback은 로그에만 저장하고 화면에는 해결버튼과 쉬운 설명을 표시한다.

## 9.13 MCP 도구 설명의 사용자 친화성

도구명은 내부적으로 안정적인 영문 식별자를 사용하되, 설명은 “언제 사용/언제 사용하지 않을지”를 포함한 자연어로 작성한다. 호스트 LLM이 범위가 불명확한 검색에서 먼저 기간을 묻게 하고, 사용자 승인 없이 배치·영구수집·파일생성을 실행하지 않도록 도구 설명과 서버 검증을 이중 적용한다.

# 12. 배치 권고 빈도에 관한 최종 정책

배치 리서치는 예외 경로이지 기본 경로가 아니다. 구현 후 벤치마크에서는 전체 대화형 검색 중 배치 승인 요청 발생률을 측정한다. 일반적인 “특정 회사 검색”과 “대표 사례 10건” 테스트 세트에서 승인 요청률이 10%를 넘으면 권고 기준이 과도한 것으로 보고 다음 순서로 조정한다.

1. 원시 후보 수가 아닌 필터 후 비교·다운로드 문서 수로 계산하는지 확인
2. DART 본문검색 후보 우선순위와 조기 종료 기준 개선
3. 41~80건 구간은 강제 확인이 아니라 결과 후 선택 안내로 유지
4. 사용자가 거절한 검색계보에서 재권고가 억제되는지 확인
5. 예상시간 모델이 실제보다 과대평가되는지 보정

전수성 표현, 5분 초과 예상, 80건 초과의 필터 후 원문 확인 또는 반복 continuation 실패가 없는 한 배치 확인창을 먼저 띄우지 않는다.

* 세션 캐시와 성능조건부 TTL 캐시
* 경량 감사 로그를 통한 검색과정 재현
* 사용자 승인 후 실행되는 중단재개 가능 배치 리서치

## 구현 전후에 반드시 검증할 부분

* DART 본문검색 자동화 접근의 허용범위와 안정성
* DART 본문검색의 실제 precision·recall
* OpenDART 원문 40건 처리의 평균·최악 응답시간
* 원문 동시성 3이 요청제한과 안정성에 미치는 영향
* TTL 디스크 캐시의 실제 쓰기 오버헤드
* 폴백 경고와 완전성 강등이 정확히 작동하는지
* 감사 로그가 검색속도에 미치는 영향
* 호스트 LLM의 법률적 분류 일관성
* MCP 클라이언트별 타임아웃과 취소지원

이번 계획의 핵심 절충은 다음과 같다.

```text
일반 질의
→ 빠르고 제한된 대화형 MCP 검색
→ 세션/짧은 TTL 캐시로 반복검색 가속

넓은 조사 필요
→ 배치 리서치 필요성·범위 사전 제시
→ 사용자 승인
→ 사용자가 선택한 실행 간격별 추가승인·체크포인트·CSV·JSON을 갖춘 배치 실행

장기 축적 필요
→ 별도 영구 인덱스 미리보기
→ 별도 사용자 승인
```

이를 통해 기존 코드의 장점인 기간분할, 원문검색, 캐시, 병렬처리 및 파일출력을 살리면서도, 사용자 승인 없는 대량수집과 무기한 저장을 방지하고 MCP의 응답시간을 통제한다.


# 15. 확정된 사용자 선택 및 권고 기본값

사용자 선택과 계획서의 권고값을 다음과 같이 확정한다.

1. **설치 형태**: 일반 설치형을 기본 배포물로 제공하고, 포터블 ZIP도 함께 제공한다.
2. **결과파일 기본 형식**: 검색 완료 후 CSV, JSON, 둘 다, 저장하지 않음 중 선택하도록 묻고, 권고값은 **둘 다**로 표시한다.
3. **배치 추가승인 기본 권고 간격**: 10분으로 한다. 사용자는 최초 승인 시 5·10·15·30분 중 선택할 수 있다.
4. **TTL 디스크 캐시**: 성능 벤치마크를 통과하면 기본 활성화한다. 벤치마크 기준을 넘지 못하면 세션 메모리 캐시만 사용한다.
5. **감사 로그 보존기간**: 30일로 한다.
6. **코드서명**: 초기 개인용 배포는 미서명으로 시작하되, SmartScreen 경고 가능성을 사용설명서에 안내한다. 다수 사용자 배포 전 코드서명을 재검토한다.
7. **DART 본문검색**: 개인용 로컬 환경에서 기술 상태가 정상이고 고정 평가질의의 검색 품질 향상이 확인되면 정책 근거 미확인 상태에서도 사용한다. 동시성 1·요청 시작간격 최소 1,000ms로 시작하고, 장애·접근거부·구조변경 또는 명시적 금지가 감지되면 자동 비활성화 후 OpenDART 원문검색으로 폴백한다.
8. **정식 지원 MCP 클라이언트**: 초기 정식 지원은 Claude Desktop로 한정한다.
9. **결과 저장 위치 UX**: 파일 저장을 선택한 경우마다 폴더 선택창을 띄워 저장 위치를 확인받는다. 마지막 저장위치는 편의를 위해 세션 중에만 기억하고, 다음 실행에서는 다시 묻는다.
10. **아이콘 콘셉트**: 귀여운 모양의 **탐정 캐릭터**로 확정한다. 작은 돋보기와 문서 요소를 함께 사용하되, 16×16 크기에서도 식별 가능한 단순한 실루엣으로 제작한다.

# 16. 포터블 ZIP 배포 방식

포터블 ZIP은 Windows에 정식 설치하지 않고, 압축을 푼 폴더 안에서 프로그램과 사용자 데이터를 함께 사용하는 배포방식이다.

## 16.1 작동 방식

```text
공시검색_MCP_포터블.zip
→ 원하는 폴더에 압축 해제
→ '공시검색 MCP 시작.exe' 또는 'MCP 등록.exe' 실행
→ OpenDART API 키 입력
→ 루트의 _local_data/.env에 저장
→ Claude Desktop MCP 설정에 현재 폴더 경로를 등록
→ Claude Desktop 재시작 후 사용
```

포터블 버전은 다음 구조를 사용한다.

```text
공시검색_MCP_포터블/
├─ 공시검색 MCP 시작.exe
├─ MCP 등록.exe
├─ 사용설명서.html
├─ app/
└─ _local_data/
   ├─ .env
   ├─ cache/
   ├─ logs/
   ├─ results/
   └─ checkpoints/
```

## 16.2 장점

* 관리자 권한 없이 사용할 수 있다.
* USB, 외장디스크 또는 다른 PC로 폴더 전체를 옮기기 쉽다.
* Windows의 설치 프로그램 목록이나 Program Files를 사용하지 않는다.
* 삭제할 때 폴더 전체를 지우면 된다.
* 개발·테스트·개인 백업에 편리하다.

## 16.3 단점과 주의사항

* 폴더를 옮기면 Claude Desktop에 등록된 실행경로가 바뀌므로 `MCP 등록.exe`를 다시 실행해야 한다.
* `_local_data/.env`에 API 키가 있으므로 폴더 전체를 다른 사람에게 전달하면 안 된다.
* ZIP을 다시 만들거나 공유할 때 `_local_data`가 포함되지 않았는지 자동 검사를 거쳐야 한다.
* 읽기 전용 위치, 네트워크 드라이브, 동기화가 불안정한 클라우드 폴더에서는 캐시와 로그 쓰기가 실패할 수 있다.
* 일반 설치형보다 자동 업데이트와 복구가 단순하지 않을 수 있다.

## 16.4 일반 설치형과의 차이

| 항목 | 일반 설치형 | 포터블 ZIP |
|---|---|---|
| 프로그램 위치 | 사용자 전용 설치경로 | 압축을 푼 폴더 |
| 사용자 데이터 | `%LOCALAPPDATA%` | 루트 `_local_data` |
| 폴더 이동 | 설치경로 유지 권장 | 이동 가능, 재등록 필요 |
| 제거 | 제거도구 사용 | 폴더 삭제 |
| 업데이트 | 설치도구가 교체·롤백 | 새 ZIP으로 app 교체 |
| 권장 대상 | 일반 사용자 | 테스트·휴대·개발 사용자 |

일반 사용자에게는 일반 설치형을 우선 안내하고, 포터블 ZIP은 별도 설치를 원하지 않거나 여러 PC에서 시험하려는 경우의 선택지로 제공한다.

## 17.2 핵심 데이터 모델

Python에서는 `dataclasses` 또는 Pydantic v2를 사용할 수 있다. MCP 경계와 파일 저장에는 Pydantic 모델을 권장한다.

<a id="REQ-SEARCH-001"></a>
### `SearchRequest`

```json
{
  "query": "공개매수 후 포괄적 주식교환 사례",
  "company": null,
  "date_from": "2018-01-01",
  "date_to": "2026-07-16",
  "target_count": 10,
  "mode": "standard",
  "max_documents": null,
  "cache_mode": "auto",
  "exhaustive": null,
  "amendment_comparison": null,
  "sequence_required": null,
  "output_mode": "interactive",
  "continuation_token": null
}
```

검증 규칙:

* `query`: 1~1,000자
* `target_count`: 1~20, 배치는 최대 500
* `max_documents`: `null` 또는 1~120. 표준 대화형 전략의 실효 상한은 40, 정정비교 전략은 기본 80·절대 120이며 사용자가 더 큰 값을 입력해도 120을 초과할 수 없다.
* 대화형 검색에서 `date_from`, `date_to`가 없고 질의에서도 확정할 수 없으면 `clarification_required` 반환
* `date_from <= date_to`
* `exhaustive=true`이면 자동 실행하지 않고 배치 미리보기로 전환
* `company`, `exhaustive`, `amendment_comparison`, `sequence_required`는 모두 선택적 힌트다. 값이 `null`이면 라우터가 `query`와 검색 맥락에서 추론하고, 호스트가 명시한 값은 추론값을 덮어쓴다.
* §8의 입력 예시는 `SearchRequest`의 축약 뷰이며, 실제 MCP 계약은 본 `SearchRequest` 하나만 사용한다.
* `mode`와 사용자 지정 `max_documents`는 전략예산의 **상한**이다. 최종예산은 `min(mode 상한, 사용자 상한, 전략 기본예산)`으로 계산하며, 정정비교처럼 더 큰 예산이 필요해도 사용자 상한을 자동 초과하지 않는다. 필요한 경우 배치 미리보기를 반환한다.

```json
  "query_variants": ["공개매수", "포괄적 주식교환"],

  "effective_document_budget": 40,

  "estimated_average_chain_length": null,
  "chain_length_estimation_basis": "not_applicable",

  "result_budget": 20,
  "first_candidate_target_seconds": 8,
```

필수 필드:


`candidate_id`는 `receipt_no`를 우선 사용한다. 동일 사건의 정정본들은 `amendment_chain_id`로 묶는다. `fulltext_match_scope`는 `body / attachment / mixed / not_applicable` 값을 가지며 DART 결과행의 `[본문]`/`[첨부문서]` 구분에서 계산한다. 동일 접수번호의 본문·첨부 행은 하나의 후보로 병합하되 행 태그는 `fulltext_row_tags[]`에 보존한다.


모든 응답은 `search_lineage_id`, `coverage`, `diagnostics`, `warnings`, `decision_summary`를 포함한다.

`search_lineage_id`는 정규화 질의 해시, 회사 식별자 및 검색기간으로 계산한다. continuation과 배치 구간 사이에서 안정적으로 유지하며, 기간범위 또는 예상 작업량이 기존보다 50% 이상 확대되면 새 lineage로 본다.

## 17.3 MCP 도구 계약

### `search_disclosure_cases`

입력: `SearchRequest`

동작:

1. 검색기간 등 필수조건 확인
2. 전략 선택 및 계획 생성
3. 대화형 예산 내 검색
4. 근거가 검증된 결과 우선 반환
5. 필요시 partial·batch 미리보기 반환

이 도구는 파일을 생성하지 않는다. 검색 완료 후 호스트가 사용자에게 파일 제공 여부와 저장 위치를 묻는다.

### `get_disclosure_evidence`

입력:

```text
receipt_no
keywords[]
include_full_preview=false
include_amendment_context=true
```

기본 반환은 근거 구간 최대 8개, 구간당 최대 500자이다. 전체 원문은 반환하지 않는다.

### `preview_batch_research`

반환:

```text
plan_id
scope
estimated_list_requests
estimated_documents
estimated_duration_range
estimated_storage
confirmation_interval_options
recommended_confirmation_interval
output_formats
```

실행은 하지 않는다.

### `run_batch_research` / `continue_batch_research`

`plan_id`와 사용자 선택 실행간격이 필요하다. 유효한 승인정보가 없으면 실행하지 않는다. 각 구간 종료 시 체크포인트를 저장하고 추가 승인을 요청한다.

`continue_batch_research`는 채널 회로차단 상태를 체크포인트에서 복원하되 `blocked_until`을 현재 벽시계로 재평가한다. 승인 대기 중 차단시간이 만료되었으면 `CIRCUIT_OPEN`으로 유지하지 않고 `PROBING` 상태에서 1회 상태확인을 수행한다.

### `export_search_results`

추가 MCP 도구로 명확히 분리한다.

입력:

```text

formats: [csv, json]
output_directory
```

사용자가 저장 위치와 형식을 확인한 이후에만 호출한다. 저장 위치가 없으면 `clarification_required`를 반환한다.

## 17.4 오케스트레이터 전략 선택 규칙

전략 선택은 우선순위가 높은 최초 일치 규칙으로 결정한다.

```text
1. exhaustive=true 또는 “전수/모두/빠짐없이” → S10
2. 정정 전후 변경내용이 핵심 → S6
3. 최종 유효본·철회 여부가 핵심 → S7
4. 두 사건 이상의 선후관계가 핵심 → S5
5. 거래소 특유 공시가 핵심 → S8
6. 특정 회사 + 명확한 공시유형 → S1
7. 특정 회사 + 본문 주제 → S2
8. 시장 전체 + 명확한 공시유형 → S4
9. 시장 전체 + 희귀 문구 → S3
10. 여러 회사의 유사 선례 → S9
```

동시에 여러 규칙이 맞으면 위 번호가 앞선 전략을 선택한다. 선택 신뢰도가 낮은 경우에도 모든 전략을 실행하지 않고, 후보 전략 두 개를 결정 로그에 남긴 뒤 비용이 낮은 전략부터 시작한다.

<a id="SUFFICIENCY-001"></a>
## 17.5 충분성 판정 공식

다음 조건을 모두 만족하면 검색을 종료한다.

```text
verified_case_count >= target_count
AND evidence_coverage_rate >= 0.9
AND duplicate_event_rate <= 0.1
AND invalid_or_withdrawn_unresolved == 0
AND requested_period_coverage_is_disclosed == true
```

`verified_case_count`와 `evidence_coverage_rate`는 **원문 검증이 완료된 확정 사례만** 대상으로 계산한다. 예비 후보는 응답에는 함께 반환하지만 충분성 공식의 분모·분자 및 종료 판정에서 제외한다.

결과가 목표보다 적더라도 다음 조건이면 부분결과로 종료할 수 있다.

```text
hard_timeout 도달
OR 문서예산 소진
OR API 제한·장애 지속
OR 남은 후보의 예상 정밀도가 임계값 미만
```

## 17.6 전략 승격 규칙

승격은 최대 2회다.

* 1차 검색에서 `verified_case_count < target_count`이면서 `verified_case_count < ceil(0.7 × target_count)`이면 사전에 지정된 2차 채널을 사용한다.
* 1차 검색에서 후보는 충분하나 원문 검증률이 낮음: 추가 채널보다 원문검증 예산 우선 배정
* DART 본문검색 장애: KIND가 사용 가능하면 후보 보강 후 OpenDART, 아니면 OpenDART 원문검색으로 직접 폴백
* 두 번 승격 후에도 목표 미달: 부분결과 반환 또는 배치 미리보기

<a id="CHANNEL-STATE-001"></a>
## 17.7 채널 상태기계

```text
HEALTHY
  ├─ 일시 오류 1회 → DEGRADED
  ├─ 구조·접근 오류 → CIRCUIT_OPEN
DEGRADED
  ├─ 정상 1회 → HEALTHY
  ├─ 연속 실패 기준 도달 → CIRCUIT_OPEN
CIRCUIT_OPEN
  └─ 차단시간 만료 → PROBING
PROBING
  ├─ 정상 → HEALTHY
  └─ 실패 → CIRCUIT_OPEN
DISABLED
  └─ 사용자 설정 또는 명백한 정책상 비활성
```

일시 오류 연속 실패 기준은 기본 2회, 구조·접근 오류는 1회다. 회로차단 상태는 프로세스 메모리에 저장하며, 배치 작업에서는 체크포인트에도 기록한다.

## 17.8 오류코드와 사용자 안내

최소 오류코드:

```text
DATE_RANGE_REQUIRED
API_KEY_MISSING
OPENDART_KEY_UNREGISTERED          # 010
OPENDART_KEY_SUSPENDED             # 011
OPENDART_IP_NOT_ALLOWED            # 012
OPENDART_NO_DATA                   # 013, 정상 결과 없음
OPENDART_FILE_NOT_FOUND            # 014
OPENDART_REQUEST_LIMIT_EXCEEDED    # 020
OPENDART_COMPANY_LIMIT_EXCEEDED    # 021
OPENDART_INVALID_FIELD_VALUE       # 100
OPENDART_IMPROPER_ACCESS           # 101
OPENDART_SERVICE_MAINTENANCE       # 800
OPENDART_UNDEFINED_ERROR           # 900
OPENDART_PRIVACY_RETENTION_EXPIRED # 901
OPENDART_HTTP_RATE_LIMITED
OPENDART_TEMPORARY_FAILURE
DART_FULLTEXT_STRUCTURE_CHANGED
DART_FULLTEXT_CIRCUIT_OPEN
KIND_SEARCH_UNAVAILABLE
DOCUMENT_PARSE_FAILED
DOCUMENT_BUDGET_EXCEEDED
SEARCH_TIMEOUT_PARTIAL
BATCH_CONFIRMATION_REQUIRED
EXPORT_LOCATION_REQUIRED
INSTALL_ROLLBACK_COMPLETED
```

API 관련 안내는 `dart_status_code`별로 분기한다.

* `010`: 입력한 키의 오타·누락을 확인하고, 등록된 키가 아니면 OpenDART에서 발급 또는 재발급한다.
* `011`: 키가 일시 중지된 상태이므로 무조건 재발급을 안내하지 않는다. OpenDART 인증키 관리와 중지사유를 먼저 확인한다.
* `012`: 현재 접속 IP가 허용되지 않는다. 키 등록 시 이용환경·IP 제한 설정과 네트워크 변경 여부를 확인한다.
* `013`: 정상적인 결과 없음이다. 오류·재시도·회로차단으로 세지 않고 빈 결과를 반환하며 채널 상태는 `HEALTHY`를 유지한다. 기간창 검색에서는 다음 창으로 계속 진행할 수 있다.
* `014`: 요청한 원문 파일이 존재하지 않는다. 동일 요청을 반복하지 않고 후보를 `document_unavailable`로 표시한다. 다른 관련 공시가 있으면 계속 진행한다.
* `020`: 요청 제한 초과다. 백오프하지 않고 즉시 중단하며 이용현황·허용시점·검색범위 축소를 안내한다.
* `021`: 조회 대상 회사를 최대 100개 이하의 묶음으로 분할하도록 계획을 재작성한다. S9와 배치 미리보기의 호출예산을 다시 계산한다.
* `100`: 코드가 허용되지 않은 필드값을 보낸 개발자 오류로 본다. 자동 재시도하지 않고 요청 파라미터를 마스킹 로그에 기록하며 해당 전략을 실패시킨다.
* `101`: 부적절한 접근이다. 반복 재시도하지 않고 접근방식·엔드포인트·요청구성을 점검하도록 안내한다.
* `800`: 시스템 점검 중이므로 재시도·키 변경을 권하지 않고 검색을 즉시 중단한다. 서비스 복구 후 다시 실행하도록 안내한다.
* `900`: 정의되지 않은 오류다. 1회만 보수적으로 재시도할 수 있으나 반복되면 즉시 중단하고 진단정보를 남긴다.
* `901`: 개인정보 보유기간 만료 상태다. 새 키 입력만으로 해결된다고 안내하지 않고 공식 안내에 따라 `opendart@fss.or.kr` 문의를 표시한다.

공통적으로 인터넷 연결, `.env` 위치·권한, 문제해결 도구의 연결검사를 확인하되 키 값은 로그·응답에 절대 포함하지 않는다.

# 19. Fable 5 개발분리 트랙

## 19.1 의미와 원칙

Fable 5는 프로그램 실행 중 호출하는 런타임 모델이 아니다. 개발 과정에서 복잡도가 높고 다수 파일에 걸친 구현·리팩터링·테스트 생성이 필요한 작업을 **별도 코딩 작업묶음**으로 분리하여 Fable 5로 개발한 뒤, 기본 개발 브랜치에 검토·병합한다.

프로덕션 코드에는 Fable 5 API, SDK, 인증키 또는 런타임 의존성을 포함하지 않는다.

## 19.2 Fable 5에 적합한 개발 작업

다음 작업을 별도 패키지 또는 브랜치로 분리한다.

### F5-1. 오케스트레이터와 10개 전략 상태기계

범위:

* `SearchRequest`→`SearchPlan` 라우팅
* 전략 우선순위·승격·충분성 판정
* 채널 상태와 예산 통합
* 결정 로그
* 10개 전략 golden test

분리 이유: 규칙 상호작용이 많고 회귀 가능성이 높아 광범위한 테스트 생성과 정적 검토가 필요하다.

### F5-2. 정정공시 체인·구조적 diff 엔진

범위:

* 원공시·정정본 연결
* 정정사항표 파싱
* 섹션·표·행 정렬
* 수치·날짜·문구 변경유형 분류
* 신뢰도 산정

분리 이유: 문서 구조가 다양하고 예외처리가 많아 고난도 파싱·비교 로직이 필요하다.

### F5-3. 사건 연결 엔진

범위:

* 공개매수 대상회사 식별
* 선행·후행 공시 관계


각 브랜치는 독립적으로 실행·테스트 가능하게 만든 뒤 순차 병합한다.

# 20. 공통 구현 계약

## 20.1 OpenDART 요청한도·호출예산 관리

OpenDART `020`은 일반 네트워크 오류나 순간적 HTTP rate limit과 구분한다. 공식 개발가이드는 이를 “요청 제한 초과”로 정의하고 일반적으로 20,000건 이상의 요청에서 발생할 수 있으나 제한은 다르게 설정될 수 있다고 안내한다. 따라서 `020`에 지수 백오프를 적용하지 않고 즉시 해당 검색을 중단한다.

구현 항목:

* `_local_data/usage/opendart_usage.json`에 API 키 해시별 호출 수, 응답 코드, 최초·최종 호출시각을 기록한다. API 키 원문은 저장하지 않는다.
* 단계 0에서 현재 공식 한도, 계산단위, 리셋경계 및 이용현황 확인 방법을 다시 검증한다. 확인 전에는 “KST 자정 일일리셋”을 확정 가정하지 않는다.
* `preview_batch_research`는 `estimated_calls`, `locally_observed_calls`, `configured_safety_budget`, `estimated_remaining_budget`을 표시한다.
* 공식 한도가 확인되면 안전여유를 둔 기본 상한을 설정한다. 확인되지 않으면 사용자가 설정 가능한 보수적 안전예산을 적용하고, 초과 예상 작업은 실행을 거부하거나 범위 축소를 요청한다.
* 오류코드를 사용자 조치별로 분리한다: 010 미등록키, 011 사용중지키, 012 접근불가 IP, 020 요청제한, 021 회사수 초과, 800 시스템점검, 901 개인정보 보유기간 만료, HTTP 429 순간 제한. 각 오류는 `dart_status_code`와 공식 메시지를 보존한다.
* `020` 안내는 “키 재발급”을 기본 해결책으로 제시하지 않고, OpenDART 이용현황 확인, 제한 해제 또는 다음 허용시점 대기, 검색범위 축소를 안내한다.

## 20.2 정정공시 체인·diff 완결성

정정공시 원문의 앞부분에 있는 정정사항표·정오표와 정정대상 보고서 정보를 **1차 연결·비교 경로**로 사용한다. 제목·회사·시점 기반 추정은 보조 경로다.

* 원공시가 요청기간 밖에 있어도 체인 완성을 위한 조회만 기간 밖으로 제한적으로 확장한다. 이 확장은 후보 모집기간을 넓히는 것이 아니며 `coverage.chain_extension`에 별도 표시한다.
* 원공시를 찾지 못하면 `chain_complete=false`, `chain_confidence=uncertain`으로 반환하고 “부분 체인 비교”임을 경고한다.
* 매칭 판정은 원공시↔최종본을 기준으로 하되 모든 중간 정정의 단계별 delta를 함께 보존한다. `change_type`에는 `cumulative_change`와 `latest_incremental_change`를 구분한다.
* 정정비교 대화형 기본 원문예산은 80건이지만 목표 20건을 보장한다고 표현하지 않는다. 예상 체인 길이·후보 탈락률을 반영한 `estimated_verified_cases`를 `SearchPlan`과 배치 미리보기 응답에 포함한다.
* 예상 체인 평균이 3개를 넘거나 검증 후보가 20건 이상이면 정정비교 전략의 대화형 예산을 최대 120건까지 허용할 수 있다. 단 사용자·mode 상한을 초과하지 않으며, 예상 5분 초과 시 배치 미리보기로 전환한다.

## 20.3 DART 본문검색 성능 튜닝

단계 0에서 기술적으로 검증된 초기 안전값은 동시성 1·요청 시작간격 최소 1,000ms다. 명시적 허용빈도는 미확인이므로 초기 제품값은 다음 A안으로 고정한다.

* A안: 동시성 1, 전역 rate limiter로 요청 시작간격 1,000ms 이상
* B안: 개인용 실험값. 동시성 1·500ms 또는 동시성 2·요청 시작간격 1,000ms 이상을 각각 별도로 측정

B안은 개인용 고정 평가질의에서 구조·접근 장애가 0건이고 검색 품질을 유지하면서 표준검색 p95를 10% 이상 줄일 때만 제한적으로 채택한다. 500ms와 동시성 2를 동시에 적용하지 않는다. 채택 후 최초 2주 관찰기간 동안 구조·접근 장애가 1회라도 발생하면 A안으로 영구 강등한다. HTTP 429 또는 일시 오류가 발생하면 즉시 A안으로 복귀하며, 강등 해제는 명시적 새 버전·재검증을 거쳐야 한다.

## 20.4 장·상호참조 안정성

중복 장 번호를 제거하고 이후 상호참조는 가능한 한 장 번호 대신 실제 정의된 고유 앵커 ID를 사용한다. `REQ-SEARCH-001`, `PLAN-SEARCH-001`, `BATCH-INTERVAL-001`, `AMEND-CHAIN-001`, `F5-CONTRACT-001`을 본문 해당 절에 정의한다. 문서 빌드 테스트에서 미정의·중복·깨진 앵커를 검사한다.

검색예산·시간·병렬성 등 수치 상수는 `config/defaults.py`와 rules 파일을 단일 출처로 하고, 본문에는 대표 값과 참조만 남긴다. 문서에 기재된 상수와 코드 기본값의 일치 여부도 같은 문서 빌드 테스트에서 검사하여, 개정 시 절 간 수치 불일치가 재발하지 않게 한다.

## 21.2 S7 최신 유효본 1차 판정

* 후보 목록에서 `rm_flags`에 `철`이 있으면 철회후보로 경고하고, 명시적 철회 문서가 연결된 경우에만 기본 결과에서 제외한다.

* I005 증권투자회사
* I006 채권공시

단계 0.5에서 I유형을 포함한 실제 목록과 공식 가이드를 대조했다. `유·코·넥·공·연`은 각각 유가증권시장·코스닥시장·코넥스시장·공정거래위원회·연결 의미의 `confirmed` 플래그다. 단계 0.6에서 I006 채권공시의 실제 `rm=채` 16건을 확보해 `채=채권상장법인` 의미도 `confirmed`로 승격했다. 다만 모두 단일문자이므로 `채`가 다른 플래그와 결합된 실제 조합 파싱은 `unconfirmed`다.

`유·코·넥`은 S8 시장소관 후보의 소프트 라우팅과 설명 필드에 사용할 수 있다. `공`은 공정거래위원회 제출·소관 신호, `연`은 연결보고 신호, `채`는 채권상장법인 신호로 별도 해석하며 서로 섞지 않는다. `rm` 누락 가능성과 조합문자, 부분통과 상태 때문에 어느 플래그도 단독으로 후보를 하드 제외하는 근거로 사용하지 않는다. `유정·코정·넥정·정연·공정`은 원문 순서를 보존해 각각의 알려진 플래그로 분해하고 회귀 fixture에서 조합 파싱을 검사한다. `채` 포함 조합은 같은 파서 계약으로 보존하되 실제 조합 표본 전까지 `combination_confidence=unconfirmed`를 표시한다. OpenDART 결과가 부족하거나 표현 기반 탐색이 필요한 경우에만 KIND를 보조 호출한다.

## 21.4 공개매수 대상회사 식별

D004 공개매수 후보에서 `corp_name != flr_nm`이면 `corp_name`을 대상회사 1차 판정값으로 사용한다. 단계 0의 10개 공시는 5개 `(corp_name, flr_nm)` 조합을 포함하므로 공시 건수와 독립 공개매수 사건 수를 분리한다. 동일 대상·제출인·공개매수 기간에 속하는 신고서·설명서·결과보고서는 하나의 사건으로 묶는다. 동일명·특수목적법인·공동제출·기간 불명확 등 예외는 보고서명과 원문으로 재검증한다. 원문을 생략한 결과에는 `target_company_source=opendart_corp_name`, `verified_from_document=false`와 신뢰도를 표시한다.

단계 0.5에서 `corp_name == flr_nm`인 8건을 원문으로 검증해 다음 보고서 역할을 고정했다.

* 공개매수신고서 → `tender_offer_filing`
* 공개매수설명서 → `tender_offer_explanatory_statement`
* 공개매수결과보고서 → `tender_offer_result`
* 공개매수에관한의견표명서 → `target_company_opinion`

목록의 `corp_name == flr_nm`만으로 자기 공개매수라고 판정하지 않는다. 의견표명서에서는 제출인인 대상회사가 `corp_name`에 나타나지만 공개매수자는 원문에 별도 회사로 기재될 수 있다. 신고서·설명서·결과보고서는 원문의 `공개매수자`와 `공개매수 대상회사명` 또는 동등한 역할 필드를 각각 추출한다. 두 정규화 회사가 같을 때만 `is_self_tender_offer=true`로 판정한다.

8건 중 더존비즈온 의견표명서는 정확한 `대상회사` 문자열은 검출되지 않았지만 원문 머리말의 `공개매수할 주식등의 발행인(의견표명자) 회사명`과 별도 공개매수자 필드로 역할을 확인했다. 이를 대상회사 불일치로 해석하지는 않되, 정확한 대상회사 필드가 없을 때는 `target_company_source=opinion_issuer_role`, `target_company_confidence=provisional`로 남긴다. 명시적 대상회사 필드를 파싱한 경우에만 `verified_from_document=true`, `target_company_confidence=high`로 올린다. 원문을 생략하면 기존처럼 `target_company_source=opendart_corp_name`, `verified_from_document=false`, `target_company_confidence=provisional`이다.

## 21.5 상세유형 사전 생성

`disclosure_terms.yaml`을 수작업 낡은 목록으로 유지하지 않는다. 공식 개발가이드 스냅샷을 기반으로 생성하고 `source_checked_at`, `source_hash`, `schema_version`을 기록한다. 공식 페이지 변경을 감지하면 테스트를 실패시키고 수동 검토 후 갱신한다.

## 21.6 정상 무데이터·추가 상태코드 계약

OpenDART 목록 API의 상태 `000`과 `013`은 모두 채널 정상으로 본다. 단계 0에서 D004 빈 기간창의 HTTP 정상 JSON `status=013`, 메시지 `조회된 데이타가 없습니다.`, 목록 부재를 확인했다. 이를 빈 배열로 정규화하고 회로차단 실패횟수에 포함하지 않으며 기간분할 검색은 다음 창으로 계속한다. 문서 API나 다른 엔드포인트의 `013`까지 실측됐다고 일반화하지 않는다. `014·100·101·900`은 실제 fixture 여부와 공식 정의·mock 근거를 구분하여 각각 문서 단위 비가용, 개발자 입력오류, 접근구성 오류, 미정의 서비스 오류로 처리한다.

## 21.7 정정체인 길이 사전추정

`estimated_average_chain_length`는 실측 전 임의 상수로 두지 않는다.

1. 목록 후보 중 `rm_flags`에 `정`이 있는 비율을 계산한다.
2. 일반 대화형 Fast Path에서는 1번 값만으로 1차 추정한다. 명시적 원접수번호·정정대상 접수번호가 있는 체인을 우선 표본추출한다. 정규화 보고서명·회사·근접기간만 같은 항목은 독립 사건일 수 있으므로 자동 병합하지 않고 `uncertain_cluster`로 계산한다.
3. 최근 완료된 동일 전략의 감사통계가 있으면 가중 평균한다.
4. 근거가 부족하면 값은 `null`로 유지하고 정정 기본예산 80을 사용한다.

SearchPlan에는 `chain_length_estimation_basis`를 함께 기록한다. 배치 미리보기 또는 첫 목록조회 후에는 실제 체인표본으로 값을 갱신한다.

## 21.9 보고서명 접두어 복수성 실측

`DisclosureCandidate`는 `report_name_prefixes[]`를 사용한다. 단계 0에서 실제 복수 접두어 조합 `[정정제출요구][기재정정]` 9건을 확인했다. 접두어 순서를 보존하고 왼쪽부터 모든 공식 접두어를 제거한 뒤 본문 보고서명을 정규화한다. 관찰되지 않은 조합도 파서는 보존하되 `unknown_prefix_combination=true`로 표시한다. `amendment_rules.yaml`에는 접두어별 의미·diff 포함 여부·공시유형 라우팅·근거 접수번호·표본 수·표본범위·신뢰도를 기록하며, `[연장결정]`은 B 계열 주요사항보고서로 라우팅한다.

`evidence_fixture`, `sample_count`, `sample_scope`, `confidence`, `checked_at`은 규칙 스키마의 필수 필드다. `sample_count < 3`인 실제 관찰 규칙은 자동으로 `confidence=provisional`이어야 하며 CI에서 누락·상향을 실패시킨다. 현재 `[연장결정]` 1건, `[정정명령부과]` 1건, `rm=철` 검증 2건은 반드시 `provisional`로 내려간다.

## 21.12 DART 본문검색 요청·응답 계약

1. 쿠키 jar를 유지하며 `/dsab007/main.do` 상태진단을 수행한다.
2. 본문검색은 동일 form으로 `/dsab007/detailSearchMain2.do`와 `/dsab007/search.ax`를 순서대로 POST한다.
3. `Referer`와 AJAX 헤더를 유지하고, 전역 rate limiter는 요청 시작시각 기준 최소 1,000ms를 보장한다.
4. 접수번호가 있는 결과행이 하나라도 있으면 `results`다. `검색건수`는 전체 결과 수이며 현재 페이지 행 수와 별도 저장한다.
5. 결과 컨테이너의 `조회 결과가 없습니다.` 또는 `검색건수: 0`은 `normal_zero`다.
6. 결과행과 0건 근거가 모두 없으면 `structure_failure_candidate`다. 상태진단 재시도에서도 반복될 때만 구조장애로 승격한다.
7. 실제 구조장애가 관찰되기 전까지 합성 fixture 결과를 운영 장애율이나 접근정책 근거로 사용하지 않는다.
8. `currentPage`는 1부터 시작한다. 단계 0.5에서 1·2페이지와 계산된 마지막 페이지를 실제 호출해 확인했지만, 결과행은 페이지 안과 페이지 사이에서 같은 접수번호를 반복할 수 있으므로 페이지 비중복을 계약으로 가정하지 않는다.
9. 단계 0.6에서 form의 `maxResults=15`, 폼 밖 `maxResultsCb=15·30·50·100`, 선택값 복사 JavaScript를 확인했다. 모든 허용값과 `maxResultsCb=100` 명시 전송에서 실제 접수번호 결과행은 10개였다. `effective_page_size` 운영 기본값은 10이며 요청값 확대에 의존하지 않는다. 응답 변화 시 `pagination_contract_changed`를 발생시킨다.
10. `검색건수`, `effective_page_size`, 페이지 링크의 마지막 `search(n)`, `ceil(search_count/effective_page_size)`를 함께 저장한다. 단계 0.5의 좁은 질의는 42건·마지막 5페이지·마지막 2행, 넓은 질의는 50,864건·마지막 5,087페이지·마지막 4행으로 일치했다.
11. 동일 쿠키 세션·동일 검색 모드에서는 검색어가 바뀌어도 모드설정 POST 없이 `/search.ax` 1회로 조회한다. 단계 0.6의 3개 검색어 전환에서 재설정 대조군과 분류·검색건수·접수번호 행 순서가 일치했다. 세션 또는 검색 모드 변경 시에만 모드설정 1회를 예상 요청 수에 포함한다.
12. 마지막 다음 페이지와 큰 범위초과 페이지는 결과행 0개와 명시적 정상 0건 응답을 반환했다. 이를 구조장애나 재시도 대상이 아니라 페이지 종료로 처리한다.
13. 후보는 질의·페이지 전체에서 `rcept_no`로 중복제거한다. 넓은 `상계 납입` 표본은 첫 페이지 내부 중복률 30%이고 1·2페이지 사이에도 `20260716000702`가 반복됐으므로 HTML 행 수를 고유 공시 수로 사용하지 않는다.
14. 한 질의의 `ceil(search_count/effective_page_size)`가 남은 결과페이지 호출예산 이하면 마지막 페이지까지 순회할 수 있다. 초과하면 질의를 먼저 정밀화하고 대표 사례는 상위 고유 후보에서 조기 종료한다. 전수성 요구는 포함 경계의 비중첩 연속 날짜창으로 나누고 창별 완료·기간 연속성·전역 `rcept_no` 합집합을 검증한 뒤, 그래도 완주가 어려우면 배치 미리보기로 전환한다.
15. 실제 DART 구조장애는 여전히 관찰되지 않았고 자동화 허용빈도도 미확인이다. 6·7번의 구조장애 분기와 개인용 조건부 사용·정책 경고는 그대로 유지한다.
16. 결과행은 시장문자, 공시유형 그룹 태그(`[발행공시]` 등), `[본문]`/`[첨부문서]` 구분, 제출인, 접수일을 구조 필드로 파싱한다. 동일 접수번호의 본문·첨부 행은 하나의 후보로 병합하되 `fulltext_match_scope`와 `fulltext_row_tags[]`를 보존하고, 본문 매치를 첨부 매치보다 우선 검증한다. 행의 유형 태그·접수일은 OpenDART 조회 전 사전필터로 사용할 수 있다.
17. 정렬 계약은 실측상 `sort=DATE`(내림차순)뿐이다. 관련도 정렬은 관찰되지 않았으므로 조기 종료 결과에는 최신순 편향을 진단정보로 표시하고, 후보 우선순위는 로컬 mechanical_score로 계산한다.
18. User-Agent는 브라우저로 위장하지 않고 단계 0 프로브와 같은 식별형 문자열(제품명/버전, 개인용 로컬 사용 표시)을 사용한다. 이는 개인용 조건부 사용 결정(9.8절)과 일관되어야 한다.
19. 단계 0.6 판정은 검색어 전환 `passed`, 페이지 크기 상향 `failed`, 날짜창 분할 `passed`다. 따라서 같은 세션·검색 모드의 변형어당 모드설정 요청은 제거하고, 실효 10행을 유지하며, 전수성 배치에서 검증된 날짜창 분할을 사용할 수 있다.

## 21.13 TLS·probe 재현성·fixture 보존

* Python 3.14/OpenSSL 조합에서 DART 인증서가 `VERIFY_X509_STRICT`와 충돌하면 해당 플래그만 완화한다. 호스트명과 인증서 체인 검증은 항상 유지하고 `verify=False`를 금지한다.
* probe는 실행별 `run_id`, PID 잠금, 최대 요청 수, 마감시간, 취소 신호를 갖고 종료 시 자식 프로세스가 남지 않도록 한다.
* raw 응답은 실행별 경로에 저장하고 성공한 실행만 원자적 manifest를 확정한다. 전역 `requests.jsonl` 무한 append 대신 실행별 요청 로그와 요약 manifest를 사용한다.
* Git에는 규칙을 재현하는 최소 golden fixture와 manifest를 유지한다. 대량 탐색 raw 응답은 압축 아카이브 또는 별도 보존경로로 분리하고 해시를 기록한다.
* `__pycache__/`, `*.pyc`, API 키, 로컬 캐시, 실행 중 lock·PID 파일은 Git에 포함하지 않는다.
* 데이터 기반 회귀 테스트는 N/Y 3사건, 재감사된 `rm`, 접두어 8종과 실제 복수 조합, `013`, D004 사건 중복, DART 정상 결과·정상 0건, API 키 비잔존을 검사한다.
* 아이엠증권 회귀 테스트는 Y 응답의 독립 접수 `20260709000043`이 비교대상 체인 최종 접수 `20260716000411`과 같은 사건으로 병합되지 않음을 명시적으로 assertion한다.
* 단계 0.5 회귀 테스트는 플래그십 6질의와 원문 분류, 좁은 질의 비중복과 넓은 질의의 실제 중복 접수번호, 세 유형 N/Y 체인, 독립사건 3표본, D004 4역할과 golden manifest 해시를 검사한다.
* 단계 0.6 회귀 테스트는 검색어 전환 3회의 완전일치, 페이지 크기 10행 유지, 날짜창 합집합·포함 경계, `rm=철` 명시 연결 9건과 오병합 반례, `rm=채` 실제 16건·조합 표본 0건, golden manifest 해시를 검사한다.
* 검색건수는 시간에 따라 바뀌므로 운영 테스트에서 42·54·13,084 같은 절대값 일치를 성공조건으로 삼지 않는다. 저장된 golden fixture 자체의 해시·파싱 결과는 고정하되 새 live probe는 양성/0건 분류, 페이지 계약, 상대적 후보 기여와 품질 변화를 비교한다.
* raw fixture 정리 전 `golden_manifest.json`에 보존 대상 파일, SHA-256, 근거 규칙, 접수번호를 기록한다. 기존 `v0.0-probe`와 과거 커밋의 raw를 제거해 실제 저장소 크기를 줄이려면 이력 재작성이나 별도 artifact 저장소가 필요하므로 사용자 명시적 승인 없는 `git filter-repo`·태그 재작성은 금지한다.

# 22. 경량성·속도 유지 원칙

본 계획은 기능이 많아졌지만, **상시 실행되는 핵심 경로는 여전히 가볍게 유지한다.** 기능 수와 실행비용을 분리한다.

## 22.1 기능 계층

### Fast Path — 대부분의 일반 검색

* 설정·API 키 확인
* 로컬 규칙 기반 전략선택
* OpenDART 목록 1차 조회
* 목록 메타데이터 필터·중복제거
* 필요한 상위 후보만 원문 검증

Fast Path에서는 KIND, 정정 diff 엔진, 사건연결 엔진, 배치 체크포인트, 디스크 캐시를 기본 실행하지 않는다.

### On-demand Path — 해당 질문에서만 로딩

* DART 본문검색
* 정정공시 체인·구조적 diff
* 공개매수 후 주식교환 등 사건연결
* KIND 보조검색
* 결과 파일 생성

각 모듈은 실제 전략이 선택되기 전에는 import·초기화하지 않는 지연 로딩을 기본으로 한다.

### Batch-only Path

* 전체 기간창 사전조사
* 모든 페이지 처리
* 대규모 체크포인트
* CSV·JSON 대량 내보내기
* 장시간 회로상태 복원

배치 기능은 사용자 승인 전 프로세스나 워커를 띄우지 않는다.

## 22.2 금지되는 무거운 구조

* 상시 데이터베이스 서버
* 백그라운드 전수 색인
* 시작 시 corp_code 외 대규모 데이터 다운로드
* 실행 중 별도 LLM 추론 호출(정규 OpenDART·DART·선택적 웹검색 네트워크 호출은 제외)
* 모든 검색채널 동시 실행
* 전체 원문의 무기한 보관
* 일반 검색에서 모든 기간창 첫 페이지 사전조사

## 22.3 성능·자원 목표

초기 권고 목표이며 단계 5에서 실측 후 확정한다.

* 유휴 메모리(기동 후 세션 문서캐시가 비어 있는 상태): 150MB 이하 권고, 200MB 하드 경고선
* 검색 중 최대 상주메모리(세션 캐시·파싱 작업 포함): 300MB 이하 권고, 400MB 하드 중단선
* 설치 후 백그라운드 상주 프로세스: 없음
* 전략 선택 p95: 50ms 이하
* corp_code 캐시가 준비된 상태의 MCP 기동 p95: 2초 이하
* 특정 회사 단순 목록검색의 첫 후보(원문 검증 전 목록 기준) p50: 3초 이하, p95: 8초 이하
* 빠른 검색 전체 검증 완료: 소프트 25초·하드 45초 이내
* 표준 검색 전체 검증 완료: 소프트 60초·하드 90초 이내
* 일반 검색에서 원문 다운로드: 로컬 기계적 점수 상위부터, 기본 최대 40건
* 로컬 감사로그 쓰기: 검색 종료 시 1회 중심
* TTL 캐시: 성능시험 통과 시에만 활성화하고 최대크기 초과 시 LRU 정리

## 22.4 기능 추가 게이트

새 기능은 다음 중 하나를 충족할 때만 Fast Path에 포함한다.

1. 동일 품질에서 외부 호출 또는 원문 다운로드를 15% 이상 줄임
2. 핵심 검색 precision·recall을 5%p 이상 높임
3. 필수 오류복구 또는 안전성을 제공함

기준을 충족하지 못하면 선택형 또는 배치 전용으로 유지한다. 이 원칙에 따라 모든 기간창을 미리 조회하는 `total_count` 사전조사는 배치·전수 검색에만 사용한다. 일반 검색에서는 어차피 조회한 첫 활성 창의 `total_count`만 무추가호출로 활용하고, 경계가 불확실할 때만 최대 2개 창을 제한적으로 표본조회한다.

## 22.5 경량성 회귀 게이트

구현 후 벤치마크에서 Fast Path의 외부 호출수·첫 후보시간·전체시간·메모리 중 어느 하나라도 기준을 지속적으로 초과하면, 최근 추가된 기능을 On-demand 또는 Batch-only로 강등한다. 10개 전략·회로차단·캐시의 일반 원칙은 앞 절의 단일 정의를 참조하며 이 절에서 중복 서술하지 않는다.
